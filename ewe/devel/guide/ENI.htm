The Ewe Native Interface

The native Ewe VM supports an interface (the Ewe Native Interface - ENI) that allows you to write dynamic-link libraries (DLLs) in C++ (or other languages) to implement native Java methods, much in the same way that Java uses Java Native Interface (JNI) for the same purpose. In fact the functionality of the ENI mirrors much that of the JNI.

 

Since your Ewe program might be working on a Ewe VM or on a true Java VM it would be convenient if you could produce a single DLL that will work both under Ewe and under Java. Luckily, because the naming convention of the ENI is different to that of the JNI, this is in fact possible. As a matter of fact, the DLLs that come standard with Ewe for Win32 (ewex_registry.dll and ewex_rapi.dll) work both under a true Java VM (but not early VMs like Microsofts Jview – these do not support JNI) and under a native Ewe VM.

 

Actually, these DLLs were produced before ENI was developed and so actually use different mechanisms to handle the native calls – however they can be easily re-written in ENI and this will be done in the future. But for the moment, it will not be helpful for you to look at the source code for these libraries since they are very different to ENI.

The object_access Class

In order to make it as easy as possible to use, ENI has been designed to allow the developer to use a single code set to implement both the Ewe and Java native methods. Then only simple automatically generated stub methods link the Ewe method implementation and the Java method implementation to the unified code.

 

The native code is unified by the use of a C++ class known as object_access. This object_access class is an abstract class that provides a uniform interface to both the Java VM and the Ewe VM. Your unified native code would then interface with the VM (and hence the objects and classes in your application) via the object_access class methods.

 

The object_access class is used for three purposes:

  1. It can be used to access general VM functions that are not particular to a specific class or object.
  2. It can be bound to a particular class and used to access information about that class (such as method and field references).
  3. It can be further bound to a particular object instance and used to access member fields and methods for that object.

 

There are two actual implementations of the object_access, called ewe_object and java_object. When the Ewe VM makes a native call, the native method would create a ewe_object and then send this to the unified native implementation code. When a Java VM makes a native call, the native method would create a java_object and then pass this to the unified code.

 

This procedure is so simple, that all the necessary functions (apart from the actual implementation of the unified code) can be generated automatically for you by the use of the NativeBuilder application now integrated into the Jewel program generator.

Native Example

Here is an example of a class with some native methods:

 

package samples.natives;
import ewe.sys.*;
 
//##################################################################
public class NativeExample{
//##################################################################
 
public static native int countCharacters(String aString);
public native String concatenate(double doubleValue,int another);
 
protected String myMessage = "Default concatenate message: ";
 
static {ewe.sys.Vm.loadLibrary("NativeExample");}
 
//===================================================================
public static void main(String args[])
//===================================================================
{
        Vm.startEwe(args);
        Vm.debug("Characters: "+countCharacters("Hello!"));
        NativeExample ne = new NativeExample();
        ne.myMessage = "Changed message: ";
        Vm.debug(ne.concatenate(1.234,56789));
        Vm.sleep(3000);
        Vm.exit(0);
}
//##################################################################
}
//##################################################################

 

We can see that this class contains two native methods. A static method, countCharacters(), and an instance (non-static) method, concatenate. The class compiles correctly, but if you attempt to run it by using: ewe samples.natives.NativeExample or jwe samples.natives.NativeExample you will get an UnsatisfiedLinkError in both cases when it attempts to call the countCharacters() method.

Next, implementations for each of the two native methods are provided, starting first with concatenate().  As mentioned before, there are two parts to the implementation of each native method; a unified implementation and a VM specific implementation. The unified implementation of the method is shown below:

 

//===================================================================
ObjectRef samples_natives_NativeExample_concatenate(object_access &obj,double par1,int par2)
//===================================================================
{
  return 0;
}

 

Note that the first parameter is a reference to a created object_access object. This is then followed by the individual parameters for method. The method returns an ObjectRef, which is a unified reference to a Java Object or Array. We will show how we can provide the correct code for this later, by replacing the “return 0;” code.

 

Following the unified implementation are the stubs for the Ewe and Java methods (which I have spaced out here for clarity):

Note the extern “C” declaration covering the two functions. This prevents name mangling and ensures the function names are exported correctly. The Ewe implementation starts with the line EWEEXPORT which tells the compiler to export the name of the function so that it can be accessed from external modules. Similarly the EWECALL can be used to specify the calling convention (which at the moment is not defined). The method itself, as you will see, begins with "Eni_". This tells the Ewe Vm that this is the function that it should call. It passes a reference to the object the method is being invoked on and then a pointer to a list of the parameters. Each parameter is a JValue union, which can be any possible Java value. The body of the function simply creates a ewe_object which is bound to the running Ewe VM (through the global variable Eni) and bound to the object on which the method is being invoked (through the parameter obj), and then passes a reference to this object, along with the decoded parameters to the unified function described before.

 

The Java implementation is similar. It also creates a java_object, bound to the JNI environment variable passed to the method (through the parameter Jni) and bound to the invoked object (through the parameter obj) and passes this along with the parameters to the unified function.

 

Lets look at the static method countCharacters()

This is very similar except that a ClassRef is passed to the functions instead of an ObjectRef. This means that the ewe_object or java_object created is bound to the NativeExample class, but not to a particular object.

 

So we can see in both cases that work only needs to be done in the functions that use the unified object_access object. The other Ewe and Java specific functions are complete and require no modifications.

 

Note: When creating native methods you must ensure that you do not overload method names - do not have two native methods (for the same class) that have the same name, but different parameter lists. If you want to provide public methods in your class with the same name, then create two private methods with different names and have the two public methods call the private ones.

Using References – Local vs Global

There are five types of references used in ENI. A ClassRef refers to a Class. An ObjectRef refers to an instance of an Object or Array. A FieldRef is used to refer to an object or class field. A MethodRef is used to refer to an object or class method. A VMRef is used to refer to the executing VM. Most of the methods in object_access will use at least one of these references. In each case, a value of 0 indicates a null reference.

 

All ObjectRef values references should only be considered valid with a single native function call from the running VM. In other words they are local – you cannot keep a reference passed to, or obtained within a native method invocation in order use it within another method invocation. If you do this you must convert the reference into a “global” reference using object_access::holdRef(Ref). This makes it available to other threads of execution and also ensures that the object is not garbage collected. Use object_access::releaseRef(Ref) to release a global reference.

 

All ObjectRef values passed to the native function or returned by any of the object_access methods are local and will be automatically released once the function returns or the object_access object is deleted. However there may be cases when you wish to release a local object earlier, so that it becomes eligible for garbage collection. This can be done by calling the freeObject() method in object_access. Once you call freeObject() you should not attempt to use that reference since it may be garbage collected at any time.

The java_string Utility Class

This C++ class is used for Unicode string handling. When you want access to a String represented by a java.lang.String object, you use the objectToJavaString() method. This method returns a pointer to a new java_string object. When you want to create a new Java String object (java.lang.String) you use the newString() method and you must provide a pointer to a java_string object which you have setup up to represent the string text.

 

You can access the array of Unicode characters in the string through the java_string::data variable, which is an array of WCHAR Unicode characters. The length of the string is given by the java_string::length variable.

 

You can create a java_string object from a Unicode array or from a char array either using the constructors:

 

java_string(const char *text, int length = -1); // text must be null-terminated if length is -1.

java_string(const WCHAR *text, int length = -1); // text must be null-terminated if length is -1.

 

java_string also provides a number of useful utilities for converting between Unicode and ASCII text.

 

char *toText() – will return a new null-terminated char array which will be a UTF8 encoded byte string. If the original Unicode characters were all English, then this is a straight ASCII text string.  You must delete the returned array when you are finished using it.

 

WCHAR *toUnicode() – will return a new null-terminated WCHAR array which is a copy of the original data. You must delete the returned array when you are finished using it.

 

java_string allows you append text to it using the cat() methods.

               java_string & cat(const char *text, int length = -1);

               java_string & cat(const WCHAR *text, int length = -1);

               java_string & catAndDelete(char *text, int length = -1);

               java_string & catAndDelete(WCHAR *text, int length = -1);

 

All these methods return a reference to itself. The catAndDelete() methods will delete the text pointers after appending a copy of the data.

 

Some convenience cat operators are also provided.

               java_string & operator << (const char *text);

               java_string & operator << (const WCHAR *text);

               java_string & operator << (java_string &js);

 

And a compare method is also provided:

               static int compare(java_string * oneStr, java_string * twoStr)

 

This compare method returns 0 if the two strings are the same, >1 if oneStr is considered greater than twoStr, and <1 if oneStr is considered less than twoStr.

Using object_access – Class/Object Independent Functions

This is the key to implementing native methods and accessing the VM and its classes and objects. Any object_access object can be used to perform class and object independent functions. These include:

Class Access Methods

            virtual ClassRef getClassRef(ObjectRef objectRef);

            virtual ClassRef getClassRef(char *className);

            virtual ClassRef getSuperclass(ClassRef clazz);

            virtual int isAssignableFrom(ClassRef clazz,ClassRef targetClazz);

 

The getClassRef() method which takes a char *className parameter requires a UTF8 encoded character string (for classes that only use English characters, this is equivalent to standard ASCII), and you should use ‘/’ to separate package names, but do not place ‘L’ before the class name, nor ‘;’ at the end.

Reference Access Methods

            virtual VMRef getVM();

            virtual ObjectRef holdRef(ObjectRef obj);

            virtual void releaseRef(ObjectRef obj);

            virtual void freeObject(ObjectRef obj);

 

ObjectRef, values are only valid for a single call to a native function. If you need to hold a reference across calls and across threads then you need to call holdRef() on the reference in order to convert it into a “global” reference that will always be valid. Call releaseRef() on the reference when you no longer need it. Use freeObject() to early-release a local object reference making it eligible for garbage collection. Do not use it again after freeing it.

Class and Object Binding Methods

            object_access & setClass(ClassRef clazz);

            object_access & setClass(char *className);

            object_access & setObject(ObjectRef obj) ;

New object_access Creation

            virtual object_access & setVM(VMRef vm);

            virtual object_access *getNew(ObjectRef forWho = 0);

            object_access *getForNewThread(int keepObjectRef = 1);

 

The getNew() method is the correct way to create a new object_access object based on an existing one. The optional forWho parameter allows the newly created object_access object to be immediately bound to a particular Java object.  You can always call setClass() on the new object_access to bind it to a class.

 

The getForNewThread() method is used to create a new object_access object which is to be used in a different thread (this actually uses the getVM()/setVM() methods to do this). However you should not use this method directly unless you have a specific reason to. Dealing with creating threads and using object_access methods across threads is dealt with using a new_thread object which is described in a later section (Performing Asynchronous Operations).

These methods are used to locate field and method references for a particular class. These references can be then used to access field values or invoke methods.

Exception Handling

Synchronization

            virtual int monitorEnter(ObjectRef monitor);

            virtual int monitorExit(ObjectRef monitor);

            ObjectRef getEweSyncObject();

            int eweMonitorEnter();

            int eweMonitorExit();

            virtual ObjectRef suspendResume(ObjectRef obj = NULL);

 

The eweMonitorEnter/Exit() methods merely call monitorEnter/Exit() using the value returned by getEweSyncObject(). This is used to synchronize a thread with the Ewe library.

 

suspendResume() is a special method that is used for two purposes. When called with a NULL or absent obj parameter, it will return a reference to the current mThread running by the VM. If the thread of execution is not being run within an mThread, this will return NULL. The value returned by this function call may then be passed to suspendResume() again as the obj parameter. In this case, a wakeup() call is made to the mThread.

 

suspendResume() is used in special circumstances that allow the Ewe application to perform a task within a native implementation that may take an extended period of time, without blocking other mThreads running in the application. This is a fairly tricky operation but is easily implemented using the new_thread C++ class provided by the ENI. This is explained in detail in a later section (Performing Asynchronous Operations).

Object and String Creation and Utilities

Method Invocation

By default it is synchronized and you should use it like this unless you have a very specific reason not to.

 

The parameters to the method must be placed in an array of JValues. The declaration for a JValue is:

 

typedef union JValue {
        int     z; //Boolean
        signed char    b; //Byte
        uint16  c; //Char
        int16   s; //Short
        int32   i; //Int
        int64   j; //Long
        float   f; //Float
        int32   l; //Object/Array
} JValue;

 

Accessing all these types can be done by simply referring to the name directly. However if you need to get/set a double value in JValue you must do this via the object_access methods getJdouble() and setJdouble(). For example:

 

        ewe_object oa(Eni);
        JValue jv1, jv2;
        jv1.i = 123; //Set an integer value.
        oa.setJdouble(&jv2, 3.1423);
        double d = oa.getJdouble(jv2);

 

The reason for this special treatment has to do with the way that the Arm version of GCC compiles on Linux systems. Using these methods ensures that the double value is stored correctly on all systems.

Performing Asynchronous Operations – Using the new_thread class

There may be times when you wish to perform a task in a native method that may take an extended amount of time, or that may call a blocking system function. However you would want to avoid blocking the calling thread, since this would block the Ewe application from running. The new_thread class is used to write code which runs in a separate thread and which is still able to access the VM resources and objects. This class then interacts with the calling Ewe thread in one of two ways:

  1. The new_thread can resume the calling thread which would have put itself into a sleep state.
  2. The new_thread can interact with a ewe.sys.Handle object allowing Ewe threads to monitor the progress of the operation.
Important Notes on Using Threads

A new_thread class represents a separate thread of execution that is unsynchronized with any other thread. Because of this fact explicit deletion of a new_thread by another thread is not advised, since it is difficult to tell precisely if the thread is still running or not. However there are some operations on new_thread objects that allow other threads to interact with it (e.g. interrupt() and join()) which means that a new_thread cannot arbitrarily delete itself when its execution ends since there may be other threads still referring to it. However never deleting a new_thread is also undesirable since it will consume system resources until deleted. Because of the garbage-collection system used in Java this is not an issue, but it is an issue in C++.

 

To solve this problem each new_thread object keeps a count of the number of references being made to the thread. As long as the reference count is non-zero the object will not be deleted. A call to ref() increases the reference count and a call to unref() decreases the reference count. When a new_thread is first created its reference count is set to 1. Note that creating a new_thread does not start it running - one of the start() methods must be used to do this. If the thread that is creating the new_thread is not interested in calling interrupt() or join() on the new_thread then it should call unref() after starting it running. After calling unref() such that the reference count is set to zero you should never call ref() again because in between the unref() and ref() call the thread could end and the new_thread object could be deleted. A new_thread will delete itself if its execution ends and its reference count is zero. If its execution ends and its reference count is not zero, deletion will be deferred until a call to unref() reduces the count to zero.

 

As mentioned above, there are two ways for a new_thread to interact with the running Ewe thread (and therefore with your application).

Method 1 - Resuming a Sleeping Calling Thread (Not Recommended)

This is a very simple method and is works as follows:

 

  1. A non-native method is used as the entry point for the blocking method.
  2. The non-native method calls the blocking native method.
  3. The native method spawns a new thread to handle the task and uses object_access::suspendResume() to get a reference to the calling mThread. It then returns immediately.
  4. The non-native method then calls mThread.waitForResume() to cause the current mThread to sleep until suspendResume() is called to re-activate it. This allows all other mThreads to continue.
  5. The new native thread handles the task, and when complete, calls object_access::suspendResume() again to wakeup the mThread.
  6. The calling mThread can then (optionally) call another native method to get the results of the method call and then continue operations.

 

This is how the Java code would look.

 

public class Calculator {
//
// This must be called from within a mThread.
//
public double doComputation(double [] bigData) throws InterruptedException
{
        nativeStartComputation(bigData);
        mThread.waitForResume();
        return computationResult;      
}
//
private native void nativeStartComputation(double [] bigData);
private double computationResult;
 
}

 

Next we look at the native implementation of the methods. This implementation will start a new thread and the original thread will return immediately. The code that executes within a new native Thread is executed in the protected void run(void *data) method of new_thread. You must override new_thread with your own class and implement the run() method.

 

This is how an implementation of the native functions may look:

 

 
class computation_thread : public new_thread
{
public:
        computation_thread(void *data) : new_thread(data){}
protected:
        //
        // This run() method is where you implement your thread -
        // it is called in a new Thread. 
        //
        void run(void *theData)
        {
               double *data = (double *)theData;
               //
               // Do the long calculation here.
               //
               // Once the run() method exits the calling mThread will be automatically
               // woken up. However you CAN explicitly wake it up by using:
               //
               // wakeupCallingThread();
               //
               double result = 0;
               //
               // The "object" member variable is a copy of the object_access object
               // that was used to start the thread via the start() method.
               //
               FieldRef f = object->getAFieldRef("computationResult","D");
               object->setDoubleField(f,result);
               delete data;
        }
};
 
void Calculator_nativeStartComputation(object_access &obj, ObjectRef bigData)
{
        if (bigData == 0) {
               obj.throwNewException(
                       "java/lang/NullPointerException",
                       "Please provide some data!");
               return;
        }
        int length = obj.arrayLength(bigData);
        double *data = new double[length];
        obj.getDoubleArrayRegion(bigData,0,length,data);
        new_thread *nt = new computation_thread(data);
        nt->start(obj);
        nt->unref();
}

 

The last three lines of code shows the creation of the computation_thread object and the method call that starts the new Thread running:

               (new computation_thread())->start(obj,data);

 

The start(object_access &obj, void *data = NULL) method is used to create a new Thread of execution. The reference to the object_access object is used by the new_thread object to create a new object_access object. This will be valid in the new Thread of execution, and will be bound to the same object as the original. This is available via the member variable object of new_thread.

 

Once the new Thread is created, the void run(void *data) method will be called. The data parameter is the optional data parameter given to the start() method used to create the Thread. Once the run() method returns the object variable will be deleted (unless you set new_thread::dontDelete to TRUE) and the mThread thread that called the original method will be woken up (unless you set new_thread::dontWakeup to TRUE).

Method 2 – Using a Handle Object (Recommended Method)

A ewe.sys.Handle object is a standard way for any thread to monitor the progress of some kind of asynchronous task. To read more about this please see the chapter on Threads and Real-Time API. This method works as follows:

  1. The native method will create a new_thread to handle the asynchronous task and it will either use a reference to an exisiting Handle object, or it will create and return a new Handle object.

  2. This Handle object is passed to the new_thread and is available for use through the handle member variable (as a pointer to a ewe_handle C++ class) in the new_thread object.

  3. The run() method in new_thread performs the task and sets the state of the Handle as it progresses.

  4. Upon return of the run() method the Stopped bit will be set and the Running bit will be cleared in the Handle flags (if this were not done already).

  5. While the asynchronous task is going on, any Ewe thread can monitor its progress using the standard Handle.wait() methods on the handle.

 

Here is an example of how this could work:

 

//===================================================================
public native Handle backgroundTask();
//===================================================================
public void testBackground()
//===================================================================
{
        ProgressBarForm pbf = new ProgressBarForm();
        pbf.showStop = true;
        pbf.execute(backgroundTask(),"Background Task");
}

 

The example shows a new method in the NativeExample.java file. It shows a native method Handle backgroundTask() and a test method testBackground(). The testBackground() method calls backgroundTask() and then passes the returned Handle to a ProgressBarForm. The ProgressBarForm would then display the progress of the task based on the information in the Handle and can optionally be used to stop the task.

 

Here is the C++ code for the implementation of backgroundTask().

 

//########################################################################
class background_thread : public new_thread {
//########################################################################
protected:     
        virtual void run(void *data)
        {
               for (float i = 0.0f; i<1.0f; i+= 0.025f){
                       Sleep(250);//Instead of doing work we just sleep.
                       if (handle->shouldStop()) {
                               handle->set(Stopped|Aborted);
                               return;
                       }       
                       handle->setProgress(i,1);
               }
               handle->setProgress(1.0f,0);
               handle->set(Succeeded);
        }
//########################################################################
};
//########################################################################
//  Implement this function: samples.natives.NativeExample.backgroundTask()
//===================================================================
ObjectRef samples_natives_NativeExample_backgroundTask(object_access &obj)
//===================================================================
{
  return (new background_thread())->startAndMakeHandle(obj);
}
//..................................................................
extern "C" {
EWEEXPORT ObjectRef EWECALL Eni_samples_natives_NativeExample_backgroundTask(ObjectRef obj,JValue * pars)
{return samples_natives_NativeExample_backgroundTask(ewe_object(Eni,obj));}
#ifndef NO_JNI_METHODS
JNIEXPORT ObjectRef JNICALL Java_samples_natives_NativeExample_backgroundTask(void *Jni,ObjectRef obj)
{return samples_natives_NativeExample_backgroundTask(java_object(Jni,obj));}
#endif //NO_JNI_METHODS
} //extern "C"
//..................................................................

 

Here you can see that, again, all the real work is done in the run() method of the background_thread object. This method basically does a countdown, adjusting the progress variable of the handle object each time. This will be reflected in the ProgressBarForm in the Java code.

 

Note that the startAndMakehandle() method by default will unreference the created new_thread and so you should not do this explicitly.

 

To start a new_thread running use one of the following three methods:

 

int start(object_access &obj, int doUnref = 0);

 

This starts the thread. Note that you must provide a valid object_access object - which will usually be the one passed to your native method. If doUnref is set to 1 then a call to unref() will also be made.

 

int startWithHandle(object_access &obj, ObjectRef theHandle, int doUnref = 0);

 

This starts the thread and provides it with an already made Handle object. If doUnref is set to 1 then a call to unref() will also be made.

 

ObjectRef startAndMakeHandle(object_access &obj, int doUnref = 1);

 

This starts the thread and tells it to create and return a new Handle that it will use. Note that by default this method will call unref() on the thread, while the other two methods do not call unref() by default.

Using the Handle of the new_thread

The handle variable used by a new_thread is a pointer to an instance of a ewe_handle C++ class. This is the access point for the associated Ewe Handle. This class has a number of methods:

 

void set(int newState)

This sets the new state of the Handle. It will automatically alert any monitoring threads to the change.

 

void setFlags(int toSet,int toClear)

This changes the flags of the state of the Handle. It will automatically alert any monitoring threads to the change.

 

void changed()

This lets any monitoring thread that some aspect of the handle has changed. The set() and setFlags() methods will automatically call this method.

 

int check()

This returns the current state of the handle.

 

void setProgress(float value, int notifyChanged = 0)

This sets the progress variable in the Handle. If notifyChanged is true, then a call to changed() will be made.

 

int shouldStop()

This checks if the shouldStop variable of the Handle has been set. This can be used as an indication to abort the task.

 

void setErrorCode(int code)

void setError(ObjectRef errorString)

void setErrorObject(ObjectRef error)

void setReturnValue(ObjectRef value)

These set the errorCode, error, errorObject and returnValue variables of the Handle.

The mThread Library

The new_thread class is actually an extension of the mThread C++ class. This, along with an mLock C++ class provides a unified multi-threaded API that works on both Win32 and Linux/Unix (using POSIX threads - pthreads). Therefore using the library ensures that your multi-threaded application will work correctly on either system.

 

The API provided by mThread/mLock is almost identical to that provided by mThread and Lock in the Ewe java library and provides the same synchronization facilities.

 

You are free to create and use mThread objects in your ENI code, but if the mThread needs to interact with the Ewe VM, then you should use a new_thread instead.

 

As mentioned before deleting mThread objects explicitly is a bad idea and so ref() and unref() calls are used instead. This same principle applies to mLock objects which also has these same two methods. When an mLock is first created it has a reference count of 1, and when its reference count is reduced to zero via unref() it will be deleted.

Creating an mThread

Just like a new_thread you create a new mThread by overriding the void run(void *data) method and possibly by overriding the default constructor. e.g.

 

class myNewThread : public mThread
{
protected:
        void run(void *data){
               // Do my work here.
               // Possibly delete the data when done.
        }
public:
        myNewThread(void *data) : mThread(data)
        {
               // Do custom initialization here, or possibly nothing.
        }
};
 

 

I can then create and start the thread like this:

 

mThread *t = new myNewThread(myData);
t->startRunning();
// At some point I will have to do this:
t->unref(); 
// Now I should not refer to t again, 
// since it could be deleted at any point.

 

Alternatively, if I am not interested in keeping a reference to the thread I could just do this:

 

(new myNewThread(myData))->startRunning(1);

 

This starts it running and dereferences it at the same time.

mThread Methods

int startRunning(int doUnref = 0);

This creates and starts a new native thread of execution. If doUnref is 1, then the reference count of the new thread will then be decremented by one. This is useful if you want to quickly create and start a new thread. e.g.:

            (new myNewThread())->startRunning(1);

This new thread will now start running and will delete itself after it is done.

 

static void nap(int milliseconds, int nanoseconds = 0);

This causes the current thread to go to sleep for a specified length of time.

 

static mThread getCurrentThread();

This returns the mThread object for the current running thread. Note: if the current running thread was not created via the mThread library then this method will cause a dummy mThread object to be created to represent the current executing thread.

 

mThread *ref();

void *unref();

These methods increase and decrease the reference count of the mThread respectively.

 

void interrupt();

This method will interrupt an mThread only if it is waiting to be notified on a lock object. Otherwise the method will have no effect.

 

int join(int timeout = FOREVER);

This method will wait for the mThread to end. If it ends within the timeout time (in milliseconds) it will return 1, otherwise it will return 0.

 

static void currentThreadEnding();

This method is used when a thread that was not created via the mThread library is ending. This method only needs to be called if:

1.      The thread was created by any method other than via the mThread library, and

2.      The thread may have used an mLock or mThread method, in which case a dummy mThread object was probably created for it.

This method will therefore delete any dummy mThread created for the thread and so free up the memory it used.

mLock Methods

The mLock is used for thread synchronization and like Ewe Lock objects they are recursive (the same thread can hold the lock multiple times) and you can wait() and notify() on them only when the current thread holds the lock.

 

void lock();

void unlock();

These methods hold and release the mLock respectively. There must be one unlock() for each lock() call.

 

int wait(int howLong = FOREVER);

This method can only be called by a thread that holds the mLock. It releases the mLock and waits for a notify() call to be made on the lock. It returns either NOTIFIED (= 1), or INTERRUPTED (= -1) or TIMEDOUT (= 0).

 

int notify(int notifyAll = 0);

int notifyAll();

These methods notify either one, notify(), or all, notifyAll(), threads that may be waiting on the mLock via wait().

 

mLock *ref();

void unref();

These methods increase/decrease the reference count of the mLock. When the reference count is decremented to zero, the lock will be deleted.
