Writing your First Ewe Program

Now let’s try to write your own Ewe program. This will simply bring up a welcome MessageBox window. Then we will try to add and display a simple bitmap within the MessageBox.

Step 1 – Create the Java File

For this example we’ll create a tests.Welcome class (i.e. a Welcome class in a tests package). First create a tests directory in the classes directory. Then create the following Welcome.java file:

 

package tests;
import ewe.ui.*;
import ewe.fx.*;
 
//##################################################################
public class Welcome{
//##################################################################
 
//===================================================================
public static void main(String args[])
//===================================================================
{
        ewe.sys.Vm.startEwe(args);
        MessageBox mb = new MessageBox(
               "Welcome",
               "Welcome to my first Program!\nI hope it works well."
               ,MessageBox.MBOK);
        mb.execute();
        ewe.sys.Vm.exit(0);
}
//##################################################################
}
//##################################################################

 

Compile this program and then run it on the command line by entering: ewe tests.Welcome or jwe tests.Welcome

Step 2 – Add an Icon to MessageBox

Modify the code to look like this:

 

        …
        mb.icon = new mImage("ewe/SmallConfig.bmp","ewe/SmallConfigMask.bmp");
        mb.execute();
        …

 

 

This has added an icon to the left of the MessageBox text.

 

The mImage class is a general-purpose graphical image rendering class that can be used to display bitmaps (represented by Image classes) with transparent images. Since a BMP file does not support a transparent color selector, images with transparent areas must be implemented using two images – the main image and a mask. The mask image should be a mono bitmap of the same size as the main image with the areas that are to be transparent in white and non-transparent images in black. The main image should have the areas that are to be transparent also in white (you can have non-transparent white areas as well). For example, the bitmaps displayed above are:

 ewe/SmallConfig.bmp  ewe/SmallConfigMask.bmp

 

There is another constructor for mImage which takes a second argument of a Color value. This value specifies that all pixels in the source image that is of the specified color should be considered transparent. So, in this case we could have done:

 

        mb.icon = new mImage("ewe/ewebig.bmp",ewe.fx.Color.White);

 

Now all pixels in the bitmap which are white will be rendered as transparent.

 

 

The bitmaps used here are all in the Ewe library, bound in with the VM. The chapter on Important GUI Issues lists all the available bitmaps in the Library and how they should be used.

Important notes on using Bitmaps and other Program Resources

In this example we accessed a few bitmaps by using their path names. When running under Java or Ewe on the PC, program resources are searched for starting in the current directory. Also, under these execution environments, file names are not case sensitive. However when running under WinCE or PocketPC all resources must be bound in a Ewe file and only such resources will be found. Additionally, resource names in a Ewe file are case sensitive and should use the ‘/’ slash as a directory separator. You should never start a resource with ‘/’ unless you are sure that it resides on a File system and not in a Ewe file.

Step 3 – Adding Your Program to the VM Launcher

So far you have been running your program using a command line prompt. However on a WindowsCE device or other PDA this is not an option. To run a Ewe application on a mobile device you have two options:

  1. Create a .ewe file for the application and place it on the device.
  2. Place the application class files on the mobile device and add a Launch Item to the VM.

 

The first option is what you would use when you are ready for full deployment of your application. It is somewhat tedious to use while you are developing the application since you have to use Jewel to update the application each time, before transferring it across to the device.

 

The second option is simpler for testing purposes. It allows you to leave your application as separate class files and resource files in the same directory structure as exists on your desktop hard drive and you simply copy across updated class files to the mobile device as you make changes on the desktop. You then use the VM Launcher (which you can leave running all the time) to execute the application.

 

To start the VM Launcher you simply run the Ewe executable without any program arguments. Then, under the Ewe menu, you select the Add Launch Item option. Then use the FileChooser to select your class file. Ensure that you specify the correct package and name for the class (tests.Welcome in this case) and you can change the title of the application. Once you do this all you need to do is click the new launch item and the application will begin running.

 

When you are doing this procedure on your mobile device, try to place your package under a directory called classes. This makes it easier for the Launcher to correctly guess what is the correct package and class name for your class.

 

Note that the VM Launcher has an option (under the Options tab) called Exit After Launch that determines whether the Launcher should exit after launching another Ewe application. This by default is true on mobile devices, but you may wish to turn this off to speed things up during testing.

Accessing Program Arguments

The programArguments member in ewe.ui.mApp is an array of strings representing command line arguments passed to the ewe VM. The command line for ewe is:

 

ewe [/<opt> [<opt_value>]] [StartClassName] [<ewefile.ewe>] [programArgument1] [programArgument2]…

 

The VM parses the command line in the following way:

  1. It starts looking for a start class name. This will be any string which does not end in “.ewe”.
  2. If it discovers any argument starting with a ‘/’ it assumes it to be a special option and will then assume that the following string is the value for that special option. You should therefore place all of these arguments before the class name OR any .ewe files.
  3. If it finds a string that does not end in .ewe it assumes it to be the starting class name and then proceeds to look for .ewe files to load.
  4. If it finds a .ewe file before finding a starting class name, it will look for a command line stored in the first (and only the first) ewe file that it finds.
  5. Once all the ewe files have been parsed and loaded it will then pass the rest of the arguments to the executing program as its program arguments.
  6. An option of "/-" tells the VM to stop processing VM options and pass all further arguments to the application.

 

The command line options are:

 

/w <screen_width_in_pixels>  = Limit the screen width.

/h <screen_height_in_pixels> = Limit the screen height.

/r  = Simulate a mobile device.

/p = Simulate a pen-based device (e.g. a PDA).

/z = Simulate a monochrome (black & white) system.

/n = Simulate a system that does not support multiple native windows.

/m = Simulate a low memory device.

/o = Rotate the screen by 90 degrees clockwise.

/O = Rotate the screen by 90 degrees anti-clockwise.

/l <locale> = Set the default Locale.

/d <directory> = Specify a specific “program directory”.

/cp <directories> = Specify a ClassPath.

/- = Force the end of VM arguments, all following arguments to be passed to the application.
