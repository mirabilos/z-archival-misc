The Auto-Installer

This chapter shows how to create a program to install your Ewe application, and possibly the Ewe VM itself, on a Windows desktop or a WinCE mobile device over ActiveSync.

 

The executable installer for Ewe applications is itself written in Ewe and so can be created by using the Jewel program creator. The process of configuring Jewel to produce the installer is not a “turn-key” (automatic) process, nor is it a very simple process. However you will only need to do the setup once for each platform, and thereafter updating the installer will simply be a case of updating a zip file, and running Jewel to re-create a ewe file and executable file.

 

The installer provides you with the following functionality.

ü      Automatic installation of the correct Ewe VM for the target device.

ü      Installation of your application, giving the user the option of selecting the install directory.

ü      Display of custom text and logos during the installation.

ü      Creation of shortcuts on the target device.

ü      Auto-execution of a target program when installation is complete.

ü      The user can choose an Express Installation which accepts all the default options and installs the VM and the Application files without requiring user input.

The Installer SDK

To get the installer tools and source code, visit www.ewesoft.com and go to the Downloads page. From there you will be able to download the installer SDK in the Ewe Development Environment section.

 

The Installer SDK consists of:

  1. Full source code of the apps.installer package (only necessary if you need to customize aspects of it – such as the language).

  2. An Installer.ewe file containing the pre-compiled installer classes and the necessary DLLs.

  3. A template configuration file.

  4. Sample installation files for a Solitaire application.

 

Creating an Installer

The installer that you will produce will be an executable file statically-linked with the Ewe VM and associated DLLs. This means that it will run on any Win32 system without the need for the Ewe VM to be installed on the system.

 

You will use Jewel to produce the installer and it will consist of:

  1. The Installer.ewe File – this contains the class file for the installer itself as well as the support DLLs.
  2. A zip file containing platform specific Ewe VMs – these can be obtained from the Ewesoft website as detailed below.
  3. A zip file containing your application files. The format for this is explained in Step 2 below.
  4. An install.cfg file. This contains options for your installation as well as specifying the display text and logo for the installer to use.
  5. Support files – including custom logos.

 

The steps for producing an installer are described in the following sections. Remember that a separate installer is necessary for each of the five supported platforms.

Step 1 – Download the Platform Specific VM Zip files from the standard VM installers.

The Installer package will look for the following zip files within its resources to determine the destination platform:

  • Ewe-PocketPC.zip – for all PocketPC platforms.
  • Ewe-Win32.zip – for all Windows desktop platforms.
  • Ewe-PalmPC.zip – for the PalmPC platforms (now obsolete).
  • Ewe-HPC.zip – for the HandHeldPC platform.
  • Ewe-BE300.zip – for the Casio BE300 platform.

 

All of these are available from www.ewesoft.com in the Downloads area – however you will note that their names on the download site will contain the version number (e.g. Ewe143-PocketPC.zip). You must rename the file to remove the version number before using it with your installer.

 

Each of these zip file has the processor specific Ewe VMs and their associated support DLLs. These zip files must be merged with your custom installer so that the latest version of the VM is installed along with your application.

 

Notes for PocketPC


There are two builds of the Ewe VM for PocketPC – a standard PocketPC build (within EweXXX-PocketPC.zip) and a PocketPC 2003 build. The standard PocketPC build will work on any PocketPC device – including any 200x devices and has no less features than the PocketPC 2003 build. So, unless you have a particular reason for using the 2003 build, you should use the standard PocketPC build.

 

Whichever build you decide to use, you must rename it to “Ewe-PocketPC.zip” before including it in your installer application.

 

Step 2 – Create a new zip file to contain your application files to install.

This zip file will contain all files that you wish to be copied to the target device during the install process. If your application contains processor/platform specific files, you will need to create a zip file for each of the platforms.

 

The zip file should contain a directory structure as follows:

ü      A directory named “installation”; this will contain the files that are to be copied into the installation directory selected by the user during the install process.

ü      A directory named “absolute” (optional); this will contain any files that are to be copied into a specific directory on the target device relative to the root (\) directory of the device – regardless of the selected installation directory.

 

The absolute directory may be left out if you are not installing any directory specific files.

 

Within both of these directories you would place the following directories:

ü      An all directory; this will contain the files that are to be copied into the installation directory for all target device processors. Usually this would include “.ewe” files and data files. This all directory may itself have a subdirectory system. This subdirectory system will be copied along with files within them into directories of the same name and structure within the target installation directory on the device.

ü      Optional mips, sh3, sh4 and arm directories; these will contain processor specific files that are to be copied into the installation directory. Usually this will be processor specific “.exe” files only. Again, these directories may themselves contain subdirectory structures.

 

If you are not installing any processor specific files on the target device, then you need not have any of the mips, sh3, sh4 or arm directories present. This also means that you will be able to use the same zip file for all platforms as well if the processor independent files are also platform independent.

 

 

 

In the example pictured above we can see that the installation directory contains only the all subdirectory. This all directory contains the processor and platform independent Solitaire.ewe file. Everything in this directory would be copied into the chosen installation directory on the target device – including any subdirectories within it. If the installation directory contained a subdirectory named mips, then all files and subdirectories within this would be copied into the target install directory if the target device was running on a MIPS processor.

Step 3 – Create an install.cfg file for each installation zip file.

This is a very important file as it specifies all of the parameters of the install – including the install zip file that you created above. The install.cfg configuration file consists of a set of lines in the format:

 

<ParameterName>=<ParameterValue>

 

For example:

 

InstallHeader=Welcome to the Solitaire Installer for %osPlatform%(TM).

 

Note that the parameter names are case sensitive. You will also notice the %osPlatform% text in the example above. This is one of the run-time variables that you can specify in the configuration file. At run-time this will be substituted with the correct value by the installer.

 

ü      All lines which start with a ‘;’ are considered to be comments and are ignored.

ü      All blank lines are ignored.

ü      If a parameter value ends with a ‘\’ then the following line is added onto the parameter value.

 

A complete template install.cfg is provided in the Installer SDK.

 

; -------------------------------------------------
; This is the installation configuration file.
; It must be named "install.cfg" (case sensitive).
; Please note that the names of the fields and variables ARE case
; sensitive.
;
; Within this file you can use special variables within String values 
; enclosed in '%' characters. e.g. %defaultInstallDirectory%
; At runtime these will have their values substituted as follows:
;
; %defaultInstallDirectory% 
; - The directory that Windows considers the "standard" directory for applications
;   on the mobile device. This value will NOT include a trailing '\'
;
; %installDirectory%
; - The actual directory the application was installed in.
;
; %eweDirectory%
; - The directory that the Ewe VM was or is installed in.
;
; %processor%
; - The processor of the mobile device, either MIPS or SH3 or SH4 or ARM.
;   When installing on the PC it is either ANSI or Unicode 
;
; %osPlatform%
; - The FULL name of the target platform. It will be one of:
; "PocketPC", "PalmPC", "HandHeld PC Pro", "Windows"
;
; %osCode%
; - The SHORT name of the target platform. It will be one of:
; "PocketPC", "PalmPC", "HPC", "Win32"
; -------------------------------------------------
 
;................................
; These affect the opening install screen.
; The logo can be any bitmap or PNG image.
; You can place \n within the InstallText and InstallHeader to denote
; a line feed. If the last character on a line is '\' then the next line
; is appended to the previous one. See the InstallText example below.
;................................
 
Logo=MyLogo.png
InstallHeader=My Application Installer - Version 1.0 for %osPlatform%(TM)
InstallWindowTitle=My Application Installer
InstallText=This program will install my application, \
Version 1.0 on your %osPlatform%(TM) mobile device.\n\nTo ensure a smooth installation, \
please disconnect your device and press continue when you have done so.
 
InstallZipFile=MyApp-Install.zip
 
;................................
; This is a special customizer class that will be created to install
; your application after the appropriate VM has been installed/updated.
; It must implement apps.installer.CustomInstall.
;................................
 
;CustomizerClass=my.class.customizer
 
;................................
; This is the default install directory for your application.
;................................
 
InstallDirectory=%defaultInstallDirectory%\MyApp
 
;................................
; These are shortcuts that you can create - you can have as many as you want.
; The format is:
;
; Shortcut=<ShortcutName.lnk>|<Target file>|<Optional Arguments>
;................................
 
Shortcut=%startMenu%\MyApp.lnk|%eweDirectory%\ewe.exe|"%installDirectory%\MyApp.ewe"
Shortcut=%startMenu%\Ewe.lnk|%eweDirectory%\ewe.exe
 
;................................
; This is the optional file to run on the mobile device after installation.
;................................
 
RunFile=%eweDirectory%\ewe.exe
RunParameters="%installDirectory%\MyApp.ewe"
 
;................................
; This is for the end message.
;................................
 
EndLogo=MyLogo.png
EndHeader=Installation Complete
EndWindowTitle=Installation Complete
EndText=The application has been installed successfully.\n\nHappy mobile computing!
 
;................................
 

 

Step 4 – Use Jewel to bring together and create the final installation executable file.

You will need to do this for each of your target platforms. To begin, start with a new Program Info file. Then do the following:

ü      Set the Program Name to the name of your choice. For example I would choose Solitaire-Win32-Setup if I was creating an installer for the Windows desktop.

ü      Set the Starting Class to be apps.installer.Installer

ü      Create a Ewe file that would contain:

1)      your platform specific install.cfg file and any supporting files (e.g. logo images),

2)      your platform specific install zip file,

3)      one of the platform specific Ewe-xxx.zip files (e.g. Ewe-Win32.zip).

 

The diagram below shows Jewel set up to create a Ewe file for creating an Installer to install Solitaire on Windows desktop.

 

 

Note that the Path in Ewe input must always be blank so that all the files added appear to be in the root directory of the Ewe file.

 

Press the Create Ewe File button to create this Ewe file. This must now be combined with the Installer.ewe file to produce the final executable installer. (You will be given a warning that the application start class "apps.installer.Installer" is not in the Ewe file. You can ignore this since this class will be in the Installer.ewe that you will be using now.) To do this switch to the Create Program File tab and then:

 

ü      Select a build target of x86 – Win32 – Static Linked (this produces an executable that does not rely on the Ewe VM being resident on the PC).

ü      If you are using a custom icon, press “Choose Icon” or “Build Icon”.

ü      Press the Add Ewe File button and then select the file Installer.ewe which would have been included in the Installer SDK.

ü      Save the program info at this point. In my example I would probably save it as Solitaire-Win32.pnf.

 

The diagram below shows the selected build target and additional Installer.ewe file selected.

 

 

Now, all you need to do is press the Build Targets button to create the final installer.  Once you have set up the .pnf file, all you will need to do from that point on to update the installer is to update your install zip file and then regenerate the .ewe file and re-build the target exe file. You may also need to update the Ewe-xxx.zip file if a newer version of the VM becomes available.

 

Note that the Target type that you choose will always be "x86-Win32-Static Linked" because the installer will always run on the Windows Desktop, even if the target is a mobile device.
