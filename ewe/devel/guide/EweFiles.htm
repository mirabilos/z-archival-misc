Ewe Files

A ewe.io.File object is very similar to a java.io.File object in that it represents a file or directory on the local file system. The API has a few differences but the most important functional difference to note is that a File object is mutable. The Ewe library does not mutate File objects used by the user, but the programmer is free to change the file/directory that a file object refers to via one of the set() method.

 

The File object can also be extended via the FileAdapter class so that it can represent data stored in a Tree other than files on a local file system. For example there is a ZipEntryFile object which treats entries in a Zip file as File objects. This allows controls like the ewe.filechooser.FileChooser to be used to browse and select entries in a zip file. However these features will be discussed in a more advanced chapter.

 

IO operations are not done on a File directly but you can call toReadableStream() and toWritableStream(boolean append) on a File to get Stream objects to be used for input and output. You can also call toRandomAccessStream(String mode) to get a RandomAccessStream for the file.

Files and Resources

A File, by default, is associated with a local file system. While this works fine when a program is running on a local device, this will not work correctly when it is being run as an Applet within a web browser. If you intend to run your program as an Applet it is important to note the following restrictions:

1.      An Applet cannot access the local file system.

2.      An Applet cannot access the file system of the web server from which it originated.

3.      An Applet can only connect via a socket to the host of the web server from which it originated.

 

Therefore, if your application is to be run as an applet there will be the issue of how to access application data. This is done via the ewe.sys.Vm.openRandomAccessStream(String path, String mode). This is a general-purpose routine that works whether the application is being run as an Applet or application, or whether it is being run from separate class files, or from a ewe file. The method works as follows:

 

1.      If the program is being run as an application it checks the local file system to see if a file exists with the specified path. If it does exist a RandomAccessFile object will be created for that file and returned.

2.      As an application, if step 1 fails, the VM will then look through all the ewe files specified when the application was run to see if it can locate the entry within them. If it locates the entry then a special RandomAccessStream object will be returned that can be used to read the entry data from the ewe file. Writing data to the ewe file will not be possible.

3.      If the program is being run as an Applet, an HTTP request will be made to the web server to try to locate the file on the server, relative to the html file that contained the Applet. If it is found the entire file is read into memory and then a MemoryFile object is returned (this provides a read/write RandomAccessStream to the file bytes in memory). Although you can write to the returned stream, it will not affect the data stored on the server.

4.      If all these fails, an IOException is thrown.

 

Note the following:

 

1.      The path that you specify should always be considered relative to the program directory (i.e. the current directory if being run via separate classes on the desktop, or the ewe file directory if being run under WinCE or desktop, or the html file which contained the Applet if being run as an Applet).

2.      Use forward slash (/) to separate directories but you should never start the path with a slash, unless you are running as an application and are specifying an absolute path.

3.      Because entries in the ewe file are case sensitive and some web servers are also case sensitive, you should consider the path to be case sensitive.

 

Using this method will ensure that your application will run on the desktop, from within a ewe file and as an Applet. This technique is used by the image creation methods, and explains how they are able to create images from a specified path regardless of being run on the desktop, from a ewe file or as an applet.

True and Simulated File Systems

A File object (but not necessarily an object that inherits from File) always represents a file on the "local" file system. Therefore these objects do not work correctly when the application is being run as an applet. However there are File objects (which generally inherit from FileAdapter) that represent other types of file systems. For example, a FakeFileSystem is used to create a simulated file system out of a read-only Zip file. When requested for a new File object, it will return a FakeFile which represents one of the files in the Zip file, or can represent a new file created by the programmer (which exists only in memory, not in persistent store). However to get an entire application to use this new file system requires some extra methods in File and requires the programmer to follow certain rules.

 

Firstly, when you want to create a new File do not use the File() constructors. Instead use one of the static method File.getNewFile() methods (this is actually a method in FileBase).

 

public static File getNewFile();

public static File getNewFile(String pathname);

public static File getNewFile(File directory, String filename);

 

These methods return a File object appropriate for the file system the VM has been set up for. By default this will be the local file system but the programmer may change this as necessary through the static ewe.sys.Vm.setFileObject(File fileObject) method.

 

Once such a File object has been retrieved you can easily traverse the file system using the methods:

 

File getParentFile();

File getChild(String childName);

File getNew(String path);

File getNew(File directory, String childName);

void set(File directory, String path);

 

The last method, set() mutates the current File to represent a different file on the file system. Very useful for traversing or iterating through directories without the overhead of creating new File objects.

File and File System Properties

There are a number of methods that you can use to determine and alter the various properties of files. These include:

 

boolean exists();

boolean isDirectory();

boolean isFile();

boolean canRead();

boolean canWrite();

int getLength();

Time getLastModified(Time destination);

 

You can create, delete, move and rename directories and files using the methods so named  see the API under File and FileBase for all these methods.

File Attributes and Permissions

Under DOS, WinCE and Unix/Linux files have attributes and permissions. Ewe allows you to view, and in some cases, modify these attributes/permissions. However since these permissions and attributes are largely incompatible (there is no archive or hidden file attributes under Unix, and no group or other permissions under DOS) attempting to access an attribute incompatible with the local file system will throw an exception. You should therefore code carefully to trap these possible conditions. This will be explained below.

 

The attributes/permissions for a File (from now on called flags) are single bit values which are bitwise OR'ed together. They are represented in the FilePermissions interface. These are DOS type attributes:

 

FLAG_ARCHIVE, FLAG_HIDDEN, FLAG_SYSTEM, FLAG_READONLY

 

and Unix type permissions:

 

OWNER_READ, OWNER_WRITE, OWNER_EXECUTE

GROUP_READ, GROUP_WRITE, GROUP_EXECUTE

OTHER_READ, OTHER_WRITE, OTHER_EXECUTE

 

The FLAG_READONLY is the only flag that will work on both type systems. Under Unix/Linux, if none of the write permissions is present, then the file is considered read-only and the FLAG_READONLY flag will be considered set. Similarly, under Unix/Linux, if you attempt to clear the FLAG_READONLY flag (that is make the file writable) then only the OWNER_WRITE permission will be enabled.

 

You determine the flags set for a file using:

 

int getPermissionsAndFlags(int interestedFlags);

 

This returns the state of the flags for a file that you specified in interestedFlags. If the running OS does not support any of the flags in interestedFlags, then an IllegalArgumentException will be thrown. You can catch this exception and possibly check for different flags.

 

You can change the flags for a file (if allowed) by using:

 

boolean changePermissionsAndFlags(int flagsToSet, int flagsToClear);

 

This method switches on and off the specified flags. It returns true if all the changes were made (or were already in effect). It throws an IllegalArgumentException if any of the specified flags are not supported on the current OS.

Other File Properties

The getInfo() and setInfo() methods are used to get/set extended information about a file or the file system. The getInfo() method has the following signature:

 

Object getInfo(int infoCode, Object sourceParameters, Object resultDestination, int options);

 

The objects used as parameters and returned by the method is dependant on the value of infoCode. The API for the INFO_XXX values used for infoCode explain what objects are to be used and returned. However the following applies in general:

1.      If null is returned, this indicates the requested information is not available.

2.      If a Throwable object is returned, this indicates an error.

3.      If a Handle is returned, this indicates the operation is progressing and will complete later. You should use the Handle to monitor the progress and then retrieve the return value from the handle using the returnValue field.

 

Values which you can get include:

 

INFO_FILE_SYSTEM, INFO_FILE_TIMES, INFO_OWNER, INFO_GROUP

 

The setInfo() method looks like this:

 

Handle setInfo(int infoCode, Object sourceParameters, int options);

 

The Handle returned is used to determine the success/failure or progress of the operation. Again, the sourceParameters object is dependant on the value of infoCode.

 

Currently the only info that can be set is INFO_OWNER/INFO_GROUP on Unix based systems.
