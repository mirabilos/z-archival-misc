Threads and Real-Time API

 

Ewe, like the original Waba, provides a single threaded user environment. This allows the Ewe VM to easily work on platforms that do not support pre-emptive multithreading. Although the VM itself may utilize multiple native threads for its operation on platforms that support this, it guarantees that only one user program thread is running at a time.

 

On top of this environment is an ewe.sys.mThread implementation that provides a threading API virtually identical to the standard java.lang.Thread. The only difference is that an mThread execution will never be pre-empted so that another can run. Instead, an mThread must explicitly call one of a number of sleep() or wait() methods to give up control to another mThread. However, because most Input/Output and GUI operations have implicit sleep() or wait() calls, you almost never have to program the sleep() yourself.

 

The mThread class uses an ewe.sys.Coroutine class for its actual implementation. However you should never create and use a Coroutine itself. Always use an mThread instead.

Non-mThread Program Threads

There are times when a thread of execution is running but is not within an mThread. These include:

 

ü      When an application begins and the VM calls the mApp.startup() method.
Since you normally use the main() method for starting your application, you will hardly have to use this thread.

ü      When a program requests timer ticks from the VM. The VM will call the ticked() method of the client object at regular intervals until the timer ticks are cancelled.
With the introduction of Coroutines, using Timer Ticks is deprecated as an mThread with sleep() methods will provide the same functionality.

ü      When a program request a callInMessageThread(). The VM will call the callback() method of the client object at the soonest opportunity.
Although this is used by the VM for OS specific functions, you will hardly have to use this method.

ü      During a paint() or resize() message from the VM.
These methods should be executed as soon as possible so calling mThread methods that cause a sleep() or wait() should be avoided in these methods in any case.

 

Each of these threads must return completely before another thread can be initiated or continued.

 

Note the the main() method that is run when a Ewe application begins is within an mThread. If you need to determine if the current thread is running in an mThread, call the static mThread method boolean inThread().

ewe.sys.mThread

The Coroutine class provides all the functionality needed for Ewe multi-threading, but does not provide an API that is consistent with java.lang.Thread. The ewe.sys.mThread class is a class that provides virtually the same API as java.lang.Thread but uses Coroutines as the underlying multi-threaded mechanism. This allows programmers to easily port multi-threaded code between Java and Ewe with very little, if any, changes to the code. Using mThread is now the preferred method of multi-threading in Ewe and you should avoid using Coroutines directly.

Suspending an mThread

Once the mThread run() method begins execution no other thread will execute until either a) the run() method exits – at which point the mThread ends permanently, or b) the mThread calls its sleep() or nap() or yield() method, or calls the join() method of another mThread. 

The GUI Event Delivery mThread

This is an mThread that is used to post Gui events from the underlying windowing system to the application Control objects. It sleeps until an event is ready for delivery and then wakes up and delivers the event via a call to postEvent() on the target control. Displaying a Frame or Form modally will cause a new event delivery mThread to be created to handle events for that Frame. This allows you to display a Form and wait for it to close even within the original event delivery mThread.

Thread Synchronization

Ewe does not support true threads and so the synchronized keyword as well as the wait() and notify() methods are not supported. Additionally, because all threads are non-preemptive the programmer does not have to worry about unexpected changes of object state by other threads unless a sleep()/join() call is explicitly made.

 

However there may be times when the programmer wants only one thread to be executing in a section of code (which may contain sleep() calls) at a time. Normally, the synchronized keyword is used for this purpose. Since Ewe does not support this keyword it instead provides a ewe.sys.Lock object for this purpose. These are explained in a later section.

TimeOut Objects

A ewe.sys.TimeOut object specifies a length of time allowed for some operation to execute with millisecond precision. You create a TimeOut object by calling its constructor with the length of time in milliseconds as the parameter. Once this is done you can call the following methods on it:

ü      boolean hasExpired() – This reports whether or not the specified time-out period has elapsed.

ü      void reset() – This tells the TimeOut to restart the countdown with the same period as before.

ü      int remaining() – This reports the number of milliseconds left before expiring. If it returns <= 0 then the timeout has expired.

ü      int elapsed() – This reports the number of milliseconds that has elapsed since creation or since the last reset().

ü      void expire() – Forces an early expiration of the TimeOut.

 

There are also two constant TimeOut objects that are predefined – TimeOut.Forever which never expires, and TimeOut.Immediate which is always expired. These are useful when you want to either wait indefinitely or not at all.

Lock Objects – ewe.sys.Lock

A Lock object allows only one thread (mThread or normal thread) to own the lock at a time. To own a Lock the boolean Lock.lock(TimeOut timeout) method must be called on the Lock. If any other thread already owns the lock the current Coroutine will wait until either a) the Lock is no longer owned and can now be owned by the waiting Coroutine, or b) the timeout object has expired.

 

If lock() returns true then the current thread now owns the Lock, otherwise it returns false and the current thread does NOT hold the lock. Once a Lock is held it must be released with a release() call on the Lock.

 

Note that if an mThread needs to wait in order to hold a Lock, it will be suspended via sleep() to allow other threads to execute. For this reason only an mThread can call the lock() method of a Lock. However there is another method called boolean grab() which attempts to hold the Lock, but will always return immediately without attempting to wait. Because no attempt is made to wait by the grab() method, it can be called by non-mThread threads.

 

Sometimes an mThread that is in possession of a Lock may wish to wait on some sort of event but still allow other threads to own the Lock until it has completed waiting. In pure Java, this is normally done through the wait() method for an object. Ewe does not support this method for all objects (as it is in Java) but it does provide a boolean Lock.waitOn(TimeOut timeout) method. This method can only be called by an mThread that is in possession of the Lock. It causes the mThread to completely give up possession of the Lock and go into a sleep state until the timeout has expired or notifyWaiting() or notifyAllWaiting() is called on the Lock. If waitOn() returns true then it has awoken due to a notify call, otherwise it has awoken due to the timeout having expired.

 

The waitOn() method does not return until after the mThread has awoken and has re-acquired ownership of the lock. notifyWaiting() will only notify one waiting mThread, while notifyAllWaiting() will notify all waiting mThreads. These methods may be called by non-mThreads because they do not involve any waiting of the calling thread.

 

Non-mThreads can call grab() to hold the Lock, any notifyWaiting() while in possession of the Lock, and the release() method to release the Lock. Non-mThread cannot call lock() or waitOn() since these both involve the thread going into a sleep state, which is only possible for mThreads.

Handles

A Handle is an object that allows you to monitor the status of an arbitrary asynchronous task. A task will set various flags and values on a handle to give an indication as to the progress and status of the running task. Any thread can monitor the progress and status of the running task by polling the handle or by waiting for certain conditions to occur.

Handle State

The state of a Handle is used to report whether the task the Handle is monitoring is running or has stopped, whether it was successful or if it failed, and when the task has attained certain critical points along its progress. The state of the Handle is really a 32-bit integer with each bit representing a different condition. The top six bits are reserved and are defined to be: Changed (indicates some kind of change of state), Running (indicates that the task has begun and is still running), Stopped (indicates that the task has stopped and no further state changes will occur), Success (indicates that the task has successfully done what it had set out to do), Failure (indicates that the task failed to do what it was supposed to), Aborted (indicates that the task was aborted due to an outside – possibly user – intervention). All other bits are free to be programmer defined.

 

A running task can change the state of a Handle by calling set(int newState) or by calling setFlags(int bitsToSet, int bitsToClear). A monitoring thread can call the check() method to check the current state of the handle. This is a non-blocking call which simply returns the current state of the handle.

 

An mThread can call one of the many waitOn()  methods to wait until certain bits of the Handle state have been set. Check the API for a description of the various waitOn() methods. An mThread can also call waitUntilStopped() to wait for the handle to report that it has stopped.

 

All the waitOn() methods will cause the calling mThread to go into a sleep() state until the appropriate bits are set. This allows other mThreads to run in the meantime.

 

A non-mThread can request a CallBack by calling callBackOnFlags(CallBack callBack, int flags, TimeOut waitFor) or  callBackOnAnyFlag(CallBack callBack, int flags, TimeOut waitFor). These will cause the CallBack object to have its callback() method executed once the appropriate state flags have been set, or when the Handle state has the Stopped bit set, or when the TimeOut has expired.

Handle Progress

The progress of a Handle is meant to represent a fractional value that indicates how close the task is to completion. A value of 0.0 indicates that the task has now begun and a value of 1.0 indicates that the task is complete. This value is only an indication of the task progress and should not be used to determine the running/stopped state of the task. During the execution of a Task the progress may be reset to zero several times due to the execution of sub-tasks within the main task.  The progress of a Handle is accessed via the progress variable.  Use of this variable is completely optional.

 

An additional variable – doing is also accessible. This is a String that can be used to give an indication as to what the task is doing at any point in time. Use of this variable is completely optional.

Other Handle methods

After changing the progress or doing fields you should call the changed() method of the Handle. This alerts any threads monitoring the Handle, that some change has occurred. You do not have to call this method after using set() or setFlags() since this automatically calls changed().

 

The void stop(int reason) method is used to request that the task be stopped. However whether or not this works depends on the Handle and the task being monitored. You should never assume that as soon as this method is called the task will stop. You should always wait until the Stopped bit has been set.

Return Values

A Handle can also be used to return a value from the asynchronous task. The task can set the returnValue variable to be any kind of object. If an error occurs in the processing it can set any of three error variables. errorCode is an int variable that can be used for a simple integer error code, error is a String variable that can be used for a String representation of the error, and errorObject is an Object variable that can represent some kind of object representation of the error – usually used to pass a Throwable/Exception object which may have been thrown during the operation.

Tasks

 

The ewe.sys.Task Interface

The ewe.sys.Task interface represents an object that performs an arbitrary asynchronous task. Usually a Task will be driven by an mThread but this is not necessarily always the case. A Task reports its progress via a Handle, which is accessible by calling getHandle() on the Task.  You can also start a Task by calling startTask() and request the task to stop by calling stopTask(int reason). Both of these return the Handle for the Task. Note that calling stopTask() does not necessarily immediately stop the task. The running task may take some time to stop its operations. After calling stopTask() you should wait until the Handle status has the Stopped flag set before assuming that the task has completely halted.

The ewe.sys.TaskObject Class

A ewe.sys.TaskObject is an implementation of a Task which you can extend to produce your own task. The TaskObject is implemented such that (assuming you want to use an mThread for executing the task) the only method you need to override is void doRun(). When startTask() is called on a TaskObject it will create a new mThread, passing the TaskObject as the Runnable object to be executed. The run() method of the TaskObject first sets the Running bit of the TaskObject Handle’s flags and then calls doRun(). Once doRun() returns it sets the Stopped bit of the TaskObject Handle’s flags.

ProgressBarForm Objects

A ProgressBarForm can be used with a Task to visually indicate the progress of the task. It uses the progress variable to draw the bar itself as the task executes. You can set the ProgressBarForm to display a cancel button and to additionally display the text of the doing variable of the Handle of the task to be monitored.

 

To display a ProgressBarForm do one of the two:

 

1.      Call setTask(Hande taskHandle, String processName) on the ProgressBarForm. This tells it what task is to be monitored, and will also prompt the form to call startTask() on the task. After calling setTask() you must then display the form by calling exec() or show() on the ProgressBarForm.

2.      Call execute(Handle taskHandle, String processName) on the ProgressBarForm (this can only be called from an mThread). This has the same effect as setTask() but will additionally display the form in a modal frame and cause the current Coroutine thread to be suspended until the Handle of the task is reported as being stopped.
