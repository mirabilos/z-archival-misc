The Control Hierarcy in Ewe

 

The GUI library in Ewe is both extensive and compact. It provides most of the features of advanced GUI libraries (such as Sun’s Swing) but with a much smaller footprint.

 

The GUI hierarchy in Ewe goes like this:

 

ewe.ui.Control – is the bottom level GUI object. This is the base object for most discrete controls such as mInput (input boxes), mButton (buttons), mCheckBox (check boxes and radio boxes) and many others. It should also be the base class to override when designing your own controls.

 

ewe.ui.Container – is the bottom level GUI container, or a Control that can contain child Controls. However Container provides only the base container functions and has no in built layout or scrolling capabilities.

 

ewe.ui.Canvas – is a Container that has the beginnings of a fully scrollable control. It has origin and virtual size information that allow it to be displayed within a scrollable area.

 

ewe.ui.CellPanel – is the most used Container in the library and is based on Canvas. A CellPanel provides all the auto-layout functionality in Ewe. This layout capability is very powerful and easy to use. So much so that no complicated sets of layout managers are provided in Ewe. They can all be handled by simple use of the CellPanel. By placing CellPanels containing controls within CellPanels, you can produce just about any layout scheme you wish without having to specify a single Control’s absolute position or size.

 

ewe.ui.Form – is another of the more used Containers in the library. A Form has the ability to display itself within a Window or Frame and thereby be rendered on the device screen. It also provides a number of useful functions, such as automatically reacting to OK, Cancel, Yes and No buttons.

 

ewe.ui.Frame – is a type of CellPanel and is a very important Control. Every control that is to be displayed on the screen must eventually be placed within a Frame. This is usually done by placing Controls in a Form and then having the Form display itself within its own Frame. You can create your own Frame and display within it your own contents, but this can be quite complicated and it is usually far more productive to use a Form instead.

 

ewe.ui.Window – is a very special Container that represents the devices native application display surface. Under Linux, Windows or WindowsCE this represents a true Window. These systems allow an application to open multiple Windows, and so Ewe allows applications on these systems to also open multiple windows. Under an operating system that does not support multiple windows (e.g. PalmOS) a Window will represent the drawing surface assigned to that application – usually the entire device screen.

 

ewe.ui.mApp – inherits from MainWindow which itself inherits from Window. There is always one mApp for your application and it provides a number of application services. Although it is a Window on multiple window systems (such as Windows and Linux) it will generally be invisible. However on single window systems (such as PalmOS) the mApp will represent the drawing area for your application and will always be visible.

 

Displaying a User Interface

 

In a nutshell, displaying a set of controls on the screen involves creating your controls and placing them within a Form, using CellPanels as general-purpose containers for organizing the controls. The Form is then displayed using one of many show(), exec() or execute() methods. As we will see later execute() is simply an exec() followed by waiting for the Form to close.

exec() vs. show() - Modal vs. Non-modal Frames

 

The exec()  and show() methods cause the Form to create a FormFrame object (a special type of Frame) and place itself within it. This new Frame is then positioned and displayed within a new Window or within an existing Window, or within an already existing Frame.

 

Please note the difference between the exec() and show() methods. The exec() method causes the frame to be displayed modally. That is, once it is displayed, all other frames will be disabled until the exec'ed frame is closed. In other words it acts like a dialog box. The show() method (which you will probably have limited use for) displays the frame non-modally - allowing other frames to still receive GUI input.

 

Please also note that unlike native Win32 or Java modal frames or dialogs, calling exec() to display a Frame will not halt execution of the current GUI message Thread. Although it will begin a new Thread to handle messages to the modal Frame, execution of the current Thread willl continue. When exec() is called, the new frame is displayed and control is returned immediately to the calling Thread. However since a new message Thread is created for the newly exec’ed Frame it is safe for calling Thread to block or go into a wait state, since the newly created message Thread will continue to service the new Frame.

 

Conversely the show() method will not create a new message Thread for the new Frame and so it is important that if the calling Thread is the old message Thread, it should not block or wait as this would block inputs to the newly showed Form.

Waiting for a Form to Close – execute() method

 

When a Form is displayed a ewe.sys.Handle object is created and assigned to the handle variable of the Form. This Handle can be used to monitor the Form so that any Thread can determine when the Form has closed. The method instance method waitUntilClosed() can be used by a Thread to wait for the Form to close. It waits until the Handle associated with the open Form has reached a Stopped state. The exit(int returnValue) or close(int returnValue) methods are usually used to cause a displayed Form to close and the waitUntilClosed() method will return the returnValue passed to these methods.

 

The Form’s execute() is simply a call to exec() followed by an immediate call to waitUntilClosed().
