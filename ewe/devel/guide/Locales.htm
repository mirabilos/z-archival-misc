Locale And Internationalization

Ewe provides a ewe.sys.Locale class that can allow your application to adapt to the locale of its runtime environment. It provides services in a number of ways:

 

1.      It can provide you with local names and values for specific items of data.

2.      It can format and parse formatted text in a locale specific way.

3.      It can allow you to automatically display the correct application specific text depending on the local language.

4.      It can correctly compare text strings of a particular language.

 

Locales in Ewe basically operate by using a main Language specifier (encoded as a two letter language code) and a Country specifier (encoded as a three letter country code). These together uniquely identify locale specific information looked up by the Locale object.

Creating a Locale Object

A Locale object created using the default constructor will represent the default language and country as specified by the operating system or Ewe installation. However you can also use ewe.sys.Vm.getLocale() to retrieve a pre-created default Locale object without having to create a new one each time.

 

Creating a Locale object for a specific language and country can be done using the constructor:

 

public Locale(String languageCode, String countryCode, boolean forceSupport);

 

This consturctor will throw an IllegalArgumentException if the language and country combination is not supported unless forceSupport is true, in which case a Locale is created that is only able to report its language code and country code as specified by the constructor parameters. Using forceSupport with an unsupported language/country is really only useful when retrieving Locale specific application data as explained in a later section.

 

The countryCode may be null, in which case one of the supported locales with the specified languageCode is used (if it exists).

 

Some platforms may be unable to report what locale the OS is running on (perhaps because the OS itself does not support locale information). However, you can set the default locale for the running application explicitly using:

 

public static boolean setDefault(String languageCode, String countryCode, int options);

 

One option available is FORCE_CREATION. This works similar to the forceSupport parameter mentioned in the previous method. It forces the support of the specified language and country even if it is not natively supported on that platform. From this point any default Locale objects created will be of the language and country specified.

 

Locales actually use a Locale ID, a platform specific integer, to internally identify locales. This ID should not be parsed but it can be used to set the locale that a Locale object represents. This can be done via the set(int id) method, or you can create a new Locale with the specified ID using the constructor Locale(int id). This is generally useful in combination with the method:

 

static int [] getAllIds(int options);

 

This method returns all the IDs supported by the running platform. You cannot determine the language and country by parsing the individual IDs, but you can create a Locale for the ID and then get information on the language and country of the Locale.

Getting Locale Specific Data

This is one of the main uses of Locale. There are a number of specific items of data that can be retrieved via the method:

 

String getString(int infoCode, int forValue, int options);

 

The API for Locale lists all the possible values of infoCode (e.g. COUNTRY_ENGLISH, DAY_OF_WEEK, etc.), what data they return and the correct values for the forValue parameter. Note that no options are as yet defined so you should set options to be zero.

 

There is another method:

 

int fromString(int infoCode, String value, int options);

 

This is used to convert from a String that represents DAY, SHORT_DAY, MONTH, SHORT_MONTH or AM_PM into the appropriate value for that data type. For example "January" should be converted to 1 when MONTH is specified as the infoCode.

Loading Locale Specific Data

This allows your application to use and display different text data depending on the locale of the application at execution time. This is achieved either through a configuration file that contains different text for different languages or through similar information stored in the Registry (the configuration file method is recommended).

 

This functionality is implemented through the ewe.sys.LocalResource interface which has two methods:

 

Object get(int resourceID, Object defaultValue);

Object get(String resourceID, Object defaultValue);

 

Basically, a LocalResource object, which internally would be linked to a particular language/country combination, would return the appropriate data object (usually a String, but potentially any object) given a resource ID code presented as an integer value or a String value.

 

You can implement your own LocalResource objects as you wish but if you use the configuration file or registry method you can use the Ewe library's default implementations.

 

To get the Ewe library LocalResource object you first create the Locale object that represents the language/country that you are interested in (in most cases this will be the system default language) and then call the method:

 

LocalResource getLocalResource(String moduleName, boolean returnNullIfNotFound)

 

This method looks for configuration information as specified by the moduleName parameter and then returns a LocalResource object which will get its data for that configuration information. If the configuration information could not be found then normally a LocalResource will be returned whose get() methods always returns the defaultValue. If returnNullIfNotFound is true, however, null will be returned if the configuration information could not be found.

 

The configuration information which is referenced by moduleName is located in one of two places, in a configuration file or in the system Registry.

Resource Configuration File

Below is an example of a configuration file.

 

{ewe.ui.Welcome}
 {LocalResources}
  {es}
  100=Bienvenido a EWE!!!
  {..}
  {es-ARG}
  100=Argentinian Bienvenido a EWE!!!
  {..}
  {es-URY}
  100=Uraguayan Bienvenido a EWE!!!
  {..}
  {de}
  100=Willkommen auf EWE!!!
  {..}
  {xx}
  100=Fake Welcome to Ewe!
  {..}
 {..}
{..}

 

The format of this file is compatible with the ewe.io.TreeConfigFile structure. This is a file that represents a tree structure, similar to a file system. The values within the {} braces represent directory names and the values of the format <name>=<value> represent individual values within the directory. The {..} values indicate the end of a directory. The structure given by the example above looks like:

 

ewe.ui.Welcome
  |
  '- LocalResources
       |
       +- es
       |   '- 100=Bienvenido a EWE!!!
       |
       +- es-ARG
       |   '- 100=Argentinian Bienvenido a EWE!!!
       |
       +- es-URY
       |   '- 100=Uraguayan Bienvenido a EWE!!!
       |
       +- de
       |   '- 100=Velkomin de EWE!!!
       |
       '- xx
           '- 100=Fake Welcome to EWE!!!
 

 

The local resources in a configuration file for a module called ModuleName must be in the structure as shown above where ModuleName replaces ewe.ui.Welcome in our example.

 

Each subdirectory under LocalResources is either a language code such as es (Spanish) or a language code combined with a country code such as es-ARG (Spanish-Argentina).

 

When searching for the value for an ID under a Locale that specifies both language and country, the LocalResource will look for a matching ID under the language-country subdirectory of the config file. If that could not be found it will look for the ID under the language subdirectory (without the country extension) of the config file. If it still could not be found it will give up and return the default value.

 

Here is an example use that goes with the config file example above:

 

        Locale l = Vm.getLocale();
        LocalResource lr = l.getLocalResource("ewe.ui.Welcome",false);
        String welcomeMessage = (String)lr.get(100,"Welcome to Ewe");

 

Note that the LocalResource object will convert an integer ID into a String ID representing that value.

Resources in System Registry

To place your locale specific data in the system registry for a particular module name you must create a key named:

 

HKEY_LOCAL_MACHINE\Software\Ewesoft\Applications\LocalResources\<moduleName>

Within this key you can have a String value ConfigFile which specifies the absolute location of a configuration file (of the same structure as explained in the previous section) that holds the local resources.

 

If you wish to place the actual resource data in the registry you must have a sub-key for each language or language-country and within that sub-key you will have data values for each ID.

Comparing/Manipulating Text Data

Depending on the locale, how text data is compared may be different. There are several methods that Locale provides for comparing text data correctly.

 

int compare(char one, char two, int options);

int compare(char [] one, int oneStart, int oneLength, char [] two, int twoStart, int twoLength, int options);

int compare(String one, String two, int options);

ewe.util.Comparer getStringComparer(int options);

 

All these methods, except the last, return a value < 0 if the first parameter is considered less than the second, > 0 if it is considered greater and 0 if both are considered equal. The options parameter can be any of IGNORE_CASE, IGNORE_SYMBOLS, IGNORE_NONSPACE ORed together (only the IGNORE_CASE option applies for the first compare()).

 

There are also methods for changing case:

 

char changeCase(char letter, boolean toUpper);

void changeCase(char [] letters, int start, int length, boolean toUpper);
