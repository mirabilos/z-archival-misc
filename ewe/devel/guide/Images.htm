The IImage Interface

 

The ewe.fx.IImage interface represents some entity that can draw itself into a Graphics context, using the various routines available in the Graphics class. This can be a bitmap, combinations of bitmaps and text, and other composite images. Some examples of classes that implement IImage are:

  • Image – this is a (usually) hardware dependant bitmapped image.
  • mImage – this is a bitmapped image along with an associated transparency mask or transparent color selector.
  • IconAndText – this is a combination of IImage and text objects laid out horizontally.
  • DrawnIcon – this is an icon that is “drawn” rather than displayed as a bit map. For example a “tick” or “cross” image.

 

Image and mImage objects are of particular importance and will be discussed in detail.

IImage Methods

IImage objects are drawn using their draw() method:

 

void draw(Graphics g, int x, int y, int options)

 

The only options defined for this method are DISABLED (indicating that the IImage should be drawn in a "gray" state) and OUTLINED. However, an IImage is free to ignore these options if it is unable to implement them.

 

You can get the dimensions of the Image using:

 

            int getHeight()

            int getWidth()

 

Graphics objects, and especially images, tend to use a lot of system resources (which is distinct from pure memory). Although these resources will be freed when these objects are garbage collected, you may wish to release the resources earlier to avoid the possibility of running out of these resources before the next garbage collection. The void free() method is used to tell the IImage to do this. You should not attempt to use an IImage after calling free() on it.

 

Some IImage objects allow you to access their pixel data directly. You can get the pixel data for a specified rectangular area within the image using:

 

int [] getPixels(int [] destination, int offset, int x, int y, int width, int height, int options)

 

The parameters x, y, width and height specify the bounds of the rectangle within the IImage of the pixel data you are interested in. Each pixel is represented by a 32-bit integer value with the Alpha-Red-Green-Blue (ARGB) format. That is: the topmost 8-bits represents an Alpha value between 0 (completely transparent) and 255 (completely opaque). The next 8-bits represent a Red value between 0 and 255 and so on. If an IImage does not support direct access to its pixels, this method will return null.

 

The values are placed in the destination array sequentially going from left to right and top to bottom within the specified rectangle. The size of the destination array should therefore be width*height+offset integers (or larger).

 

The method boolean hasAlpha() returns whether or not the alpha channel is used by this IImage. If it is not used then the alpha values in the image pixel data should be ignored (or taken to be 255 – completely opaque).

 

Some images have a background color associated with them. The method Color getBackground() can be used to get this color, if it is specified. Note that when you draw an IImage, it will not fill in its background color – the background color is merely a suggestion for display.

The Image Class

An Image object represents a bitmapped image loaded into memory, or a bitmapped area that can be drawn to. You can either use an Image as drawing surface or you can load a predefined bitmapped image from the local file system or a sequence of formatted bits. You should not draw onto an Image that has been decoded from a formatted file – this will fail on most systems.

Formatted Bitmaps (Pictures)

Ewe natively recognizes three image formats: Windows Bitmap (BMP) images, Portable Network Graphics (PNG) images and JPEG images. For BMP images Ewe is able to decode uncompressed 2, 16 and 256 indexed color images as well as 24-bit true color images. For PNG images, Ewe is able to decode all PNG image formats, compressed or not, interlaced or non-interlaced. Ewe is able to decode JPEG RGB and grayscale images.

Loading Image Resources

There are three methods for decoding formatted images, implemented by several constructor methods described below. If an error occurs during the creation of an Image, or if the format is not decodable for any reason, these constructors will throw an IllegalArgumentException.

Loading from a Named Resource

You can create an Image using the constructors:

 

Image(String path)

Image(String path, int options)

 

The first constructor simply calls the second with options set to 0. The image creation options are discussed later.

 

Both these constructors first attempt to read all the bytes as specified by the path via the ewe.sys.Vm.loadResourceOrFile(String path) method. This method loads the bytes of a resource completely into a ByteArray object. The named resource may be in a file on the local file system, or stored in one of the Ewe files associated with the application, or from the source server if the program is running as an applet. Once the bytes have been read into the ByteArray the image is decoded if it is of an acceptable format.

 

Note that when loading images this way you should not use absolute path names, but rather a path that is relative to the current working directory. For example: Image(“myImages/Smile.bmp”) is good, but Image(“C:/ewe/myImages/Smile.bmp”) is bad. Also note that when an image is loaded from a Win32/WinCE file system it is not case sensitive, but when being loaded from a Ewe file or from some web servers and some file systems, it is case sensitive – so ensure that you have the case correct to begin with.

Loading from a ByteArray or Stream

You can also create an image from an array of bytes (which must contain a complete formatted BMP, PNG or JPEG image) via the ewe.util.ByteArray object and the constructors:

 

Image(ByteArray bytes, int options)

Image(ByteArray bytes, int options, int requestedWidth, int requestedHeight)

Image(ByteArray bytes, int options, Rect sourceArea, int requestedWidth, int requestedHeight)

 

You can also create an image from an input stream (ewe.io.Stream) using the constructors:

 

            Image(Stream bytes, int options)

Image(Stream bytes, int options, int requestedWidth, int requestedHeight)

Image(Stream bytes, int options, Rect sourceArea, int requestedWidth, int requestedHeight)

Decoding an Image at a Different Size

The requestedWidth and requestedHeight parameters in the constructors above are used to request that the Image be created at a particular size (usually smaller than the image's actual size). The purpose of this is generally to allow a smaller version of an image to be created (i.e. a thumbnail) from the full sized image. This is useful if the original image is very large when decoded and would take up a large amount of memory (e.g. a 1600x1200 JPEG image will take up nearly 8MB of memory when decoded at full size).

 

When specifying requestedWidth and requestedHeight, by default the image will be scaled so as to fit in the specified width and height, but scaled equally horizontally and vertically so as to maintain its aspect ratio. Therefore the actual width and height of the decoded image may differ from the requestedWidth and requestedHeight specified. However if the Image.SCALE_IMAGE option is used with the options parameter, then the image width and height will be scaled independently to fit exactly into the requestedWidth and requestedHeight.

 

However not all versions of the VMs support scaling during decoding and some only support scaling for selected image formats. In fact, scaling during decoding is only currently available for JPEG images running on a native VM on Win32, WinCE and Linux (but not on the Java Ewe VM). However you should never make this assumption at run-time. You should always code for the fact that the scaling may not be supported. The section on "Checking the Image Before Decoding" will tell you how to determine if this scaling is possible before decoding the image and how to decide how to handle situations when scaling during decoding is not possible.

 

If scaling during decoding is not supported then an IllegalArgumentException will be thrown. In this case you should decode the Image at full size and then use a ewe.fx.PixelBuffer object to scale the image up or down to the size that you want (see the next chapter for details on this).

Decoding a Portion of an Image

The sourceArea parameter in the constructors above are used to request that only a portion of the Image be decoded, and that it should be decoded to the size specified by requestedWidth and requestedHeight. Again, this is very useful when you need to display a portion of a very large image on a low memory device.

 

The same rules applies as when decoding the full image at a different size. Specifically: it is only supported for certain image types on certain platforms and an IllegalArgumentException will be thrown if this constructor is called and the VM does not support scaled decoding.

 

Note that the sourceArea must represent a rectangle that is completely within the image otherwise an IllegalArgumentException will be thrown.

Checking the Image Before Decoding

If you need to determine the image size, or determine whether it can be scaled during decoding (and thereby reduce its memory usage), then you can call one of the two static getImageInfo() methods.

 

static ImageInfo getImageInfo(ByteArray imageBytes, ImageInfo destination)

static ImageInfo getImageInfo(Stream imageBytes, ImageInfo destination)

 

These methods decode just the amount of bytes required to determine the image format and its size. This information is returned in an ImageInfo object which currently gives the following information:

 

int width – the width of the image.

int height – the height of the image.

int size – the approximate number of bytes required for the image when decoded.

boolean canScale – whether the image can be scaled during decoding.

int format – the format type of the Image (one of: FORMAT_PNG, FORMAT_JPEG, FORMAT_BMP).

 

Based on this information you can determine how to decode the image – or whether to decode it at all given the system's memory constraints.

 

However using this method can present a problem when used with an encoded image in a Stream that is not a RandomAccessStream. This is because the getImageInfo() method will read a number of bytes from the stream, but the constructor that will decode the image from the Stream will also need to read these same bytes during decoding. If the Stream provided cannot seek() back to the beginning (if it is not a RandomAccessStream) then the bytes already read in by getImageInfo() will not available to the constructor for image decoding.

 

This is solved by using the ewe.io.RewindableStream class. A RewindableStream adapts a standard input Stream and makes it "rewindable" by buffering and storing all data read until a rewind() method is called. The rewind() method tells the stream to provide data from the start of the stream (actually read from the buffer) until all the buffered data (i.e. the data read before rewind() was called) has been re-read. After that point it will provide data directly from the source Stream.

 

Since this is not necessary for a RandomAccessStream the RewindableStream provides a static utility function toRewindableStream(Stream input) which will either return a new RewindableStream or it will return the provided input Stream if the input Stream is already a RandomAccessStream. Another static utility function rewind(RandomAccessStream input) is used to rewind the input stream if it is a RewindableStream, otherwise it will simply call seek(0) if the provided Stream is a true RandomAccessStream.

 

Some code may help to demonstrate this:

 

package tests;
import ewe.ui.*;
import ewe.io.*;
import ewe.fx.*;
import ewe.filechooser.*;
 
//##################################################################
public class TestImage{
//##################################################################
//===================================================================
public static boolean displayImage(Stream image) throws IOException
//===================================================================
{
        // If the "image" stream is a RandomAccessStream 
        // (but not itself a RewindableStream) then "stream" will
        // be set to the "image" stream itself. Otherwise a new RewindableStream
        // is created for the "image" stream and assigned to "stream".
        //
        RandomAccessStream stream = RewindableStream.toRewindableStream(image);
        //
        ImageInfo ii = Image.getImageInfo(stream,null);
        if (ii.size > 500000 && !ii.canScale){
               new MessageBox("Image too large","That image is too large to display!",0)
                       .execute();
                return false;
        }
        // If "stream" is a true RandomAccessStream, this method will simply
        // call seek(0) on it. Otherwise it will invoke the "rewind()" method
        // on the stream.
        //
        RewindableStream.rewind(stream);
        //
        Image i = ii.canScale ? new Image(stream,0,200,200) : new Image(stream,0);
        Form f = new Form();
        f.addLast(new ImageControl(i));
        f.title = "Image";
        f.execute();
        return true;
}
//===================================================================
public static void main(String args[])
//===================================================================
{
        ewe.sys.Vm.startEwe(args);
        while(true){
                ImageFileChooser fc = new ImageFileChooser();
               if (fc.execute() == fc.IDCANCEL) break;
               File got = fc.getChosenFile();
                Stream in = null;
               try{
                       in = got.toReadableStream();
                       displayImage(in);               
                }catch(Exception e){
                       new ReportException(e,"There was an error creating the image.",null,false)
                            .execute();
                }finally{
                       if (in != null) 
                            in.close();
               }
        }
        ewe.sys.Vm.exit(0);
}

 

Creating Images for Drawing

You can create an image for use as an off-screen drawing surface by simply calling the constructor with the specified width and height: Image(int width, int height, int options).  You can then create a Graphics object for the image by using the Graphics constructor Graphics(ISurface surface) since Image implements the ISurface interface.

 

Note that an image created in this way is not necessarily “blank” – you should clear the image using Graphics.fillRect() if you need to before drawing on it.

 

If you intend to perform graphics operations where the Alpha channel is used, you should call the enableAlpha() method immediately after creating the image. Note that alpha values cannot be written using Graphics drawing primitives. If you are going to be composing images that contain pixels with different alpha values, you must use the PixelBuffer class. This is explained in the following chapter.

Copying Images

The correct way to create a copy of an Image is to use the constructor: Image(Image other, int options). This ensures that the image pixel manipulation routines outlined below will work on all platforms (e.g. Java 1.1).

Manipulating Image Pixels

You can access the individual color values of the pixels of an Image using the getPixels() and setPixels() methods. However for this to work you may have to create the image with the TRUE_COLOR option. The signature for the two methods are similar:

 

int [] getPixels(int[] dest, int offset, int x, int y, int width, int height, int options);

int [] setPixels(int[] source, int offset, int x, int y, int width, int height, int options);

 

The Grahics.drawImage() Method

This method is used to draw an Image into a Graphics context. However this method will not correctly draw Images with alpha channels. You must set the drawing op of the Graphics object to be DRAW_ALPHA first before calling the drawImage method, and then reset it to its original value.

 

To avoid going through this, always use an mImage (explained below) created from an Image with an alpha channel when drawing images. This will always draw the Image correctly regardless of what method it employs to implement transparent areas.

 

Wherever possible, use PNG images for your application images. They are smaller than BMP images and can contain transparency information within them. If you are creating icons for your application you can easily convert them to PNG images with transparent areas by doing the following:

 

  1. Create your icon as a BMP image, filling the transparent areas with a color that is not used in any of the opaque areas of the icon.
  2. Use Ewe to run the ewe.graphics.PngIconMaker class.
  3. Use this application to load your icons, specify the transparent color in the icon, and then save the icon as a PNG image.

The mImage Class

 

mImage can be thought of as an advanced read-only image class. It is used for three purposes:

  1. To allow Images to be drawn with transparent pixels either using an Image's own alpha channel or using a transparency mask (which is a monochrome Image) or using a selected transparent color.

  2. To allow images to be placed in some sort of organized context where its position is variable (e.g. in a Mosaic object). The location field is a Rect object that specifies the images x and y co-ordinates as well as its width and height.

  3. To produce an IImage implementation by extending mImage and simply overriding the doDraw() method in addition to setting the width and height fields in the location field.

 

An mImage usually contains an Image object sometimes coupled with a transparent color or mask image to specify transparent areas of the Image. If the Image object was decoded from a PNG formatted image, then the transparent color or mask is not necessary, since transparency information can be included in the PNG data.

Transparent Images using A Mask Image or Transparent Color

These were the original methods that Ewe used to display images with transparent areas, before PNG and alpha-blending support was added. Specifying a Transparent Color for an Image simply specifies that all pixels with that particular color should be considered completely transparent.

 

A Mask for an Image should be a monochrome bitmap of exactly the same size as the original image, with all opaque pixels being black, and all transparent pixels being white. Additionally, the pixels in the original image that are to be transparent should also be white, the opaque pixels can be any color. Again, this method will only specify completely opaque or completely transparent.

 

Creating an mImage specifying a Mask or transparent color can be done using the constructors below:

 

mImage(String imagePath, String maskPath);

mImage(String imagePath, Color transparentColor);

 

You can also create an mImage from an Image (in fact any IImage) directly, and then set the Image mask or transparent color afterwards using the constructor:

 

            mImage(IImage drawable);

 

So for example you could do this:

 

        Image bitmap = new Image(bytesInByteArray,0);
        Image mask = new Image(maskBytesInByteArray,0);
        mImage mi = new mImage(bitmap);
        mi.mask = mask;

 

or this:

 

        Image bitmap = new Image(bytesInByteArray,0);
        mImage mi = new mImage(bitmap);
        mi.transparentColor = new Color(255,0,0);

 

Transparent Images using an Alpha Channel

If an Image has an alpha channel, then you can create an mImage from it directly using the constructor mentioned before:

 

            mImage(IImage drawable);

            mImage(IImage drawable, int drawMode);

 

There is no need to set a Mask or Transparent Color in this case. However when this method is used with an Image object that contains an alpha channel and the drawMode is not explicitly Graphics.DRAW_ALPHA, the following is done:

  1. The individual Image pixels are checked – if there are pixels that have an alpha value between 1 and 254 (i.e. not fully opaque and not fully transparent) then an alpha blending draw mode will be used for drawing the image.

  2. If all the pixels are either completely opaque (255) or completely transparent (0) then a monchrome bitmapped mask is created for the image and a modified version of the image is used.

 

Why is this done? Although DRAW_ALPHA works with fully opaque or fully transparent images, it is considerably slower on some platforms than the Image and Mask combination. Therefore, whenever it can, mImage will resort to this method for drawing images that have either fully opaque or fully transparent pixels.

 

This operation is transparent to you but there is one effect of this. Since the original image is not used after the mImage has been set up, its resources are not needed. You should therefore free the original image by calling freeSource() on the mImage to tell it to free the original Image if it is not being used directly by the mImage. So if you create an mImage from an Image using one of the two constructors above and you are not going to use the Image for any other purpose in your program, then you should call freeSource() to tell the mImage that it can free the source image if it is not using it.

 

        Image bitmap = new Image("MyImage.png");
        mImage mi = new mImage(bitmap);
        mi.freeSource(); // Do this if you do not use "bitmap" anywhere else.

 

Displaying Images – the ImageControl Control

Drawing an image on a drawing surface is simply a case of using its draw() method. However if you want to display an image on a user interface screen it is easier to use an ImageControl object.

 

An ImageControl is used to display a single IImage although the image that is being displayed can be changed dynamically using the setImage() method. The image is always centered in the control but there are options available to determine how an image that is of a different size to the control is handled. You do this by setting the options variable to be a combination of the following:

 

  • SHRINK – this tells the control to shrink an image to fit within it if the image is too large.
  • STRETCH – this tells the control to stretch an image to fit within it if the image is too large.
  • FIT – Used in combination with either or both of the two above, this tells the control not to change the aspect ratio of the image. The image is either shrunk or stretched such that one dimension fits exactly into the control and the other dimension is smaller than or equal to that of the control. This results in the entire image being visible with its aspect ratio preserved, although there may be empty space along one of the dimensions.

 

By default, the options is set to SHRINK|STRETCH|FIT.
