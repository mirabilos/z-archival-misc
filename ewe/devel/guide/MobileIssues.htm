Mobile Application Issues

There are certain issues that affect mobile applications that do not arise when running desktop applications. These are discussed below. Note that dealing with many of these issues usually requires that your application override ewe.ui.mApp – in which case you should override the run() method and start your applications from that point.

Application Closing

On mobile operating systems like Windows CE (and PocketPC) applications may be closed by the system when memory becomes low. On WindowsCE this is through a WM_CLOSE message being sent to the application. This is precisely the same message that is sent when the close button (‘X’) is pressed on a native Win32 window. It is therefore not possible to tell whether the message was generated by the user or by the system. Further, if the application does not terminate itself in response to the WM_CLOSE message within a short while, the application will be forcibly terminated by the system – causing a possible loss of open data. Therefore, ignoring this message is not an option.

 

Applications that do not need to save data or state before exiting do not need to worry about this issue. However with applications that do need to save data or state the best strategy to adopt is this:

 

1.      Do not display windows that have a Close button in the title bar. Instead, provide a Ewe close button within the displayed Form or provide a Close menu option. This way, you know that these controls can only be triggered by user input and you can then act accordingly. To show a window without a close button use the line:
windowFlagsToClear |= Window.FLAG_HAS_CLOSE_BUTTON;
in your Form.

2.      If a Ewe application is running on a MOBILE device and it receives a WM_CLOSE message, then the closeMobileApp(Window targetWindow, int flags) method of the application main mApp class will be called. Your application should override mApp and then override this method to save state and exit unconditionally using exit().  Note that on some systems it may be possible for the system to tell the difference between a user requested close and a system requested close. If this is the case, then the flags parameter will have the WindowEvent.FLAG_CLOSE_BY_USER bit set.

 

closeMobileApp(), by default, will exit the application if the targetWindow does not have a Close button in the title and if the flags parameter does not have the WindowEvent.FLAG_CLOSE_BY_USER bit set. If the targetWindow does have a Close button or if the WindowEvent.FLAG_CLOSE_BY_USER bit is set in the flags parameter, then the application will not be exited. This means that small MessageBoxes that you may display will not exit the application when closed by the user.

 

You should not attempt to display any windows or interact with the user in closeMobileApp() since the OS may terminate the application forcibly within a second or so in any case. You should simply save the application state as quickly as possible and then exit the application.

File Access

Under WindowsCE there is no such concept as a “Current Working Directory”. Therefore all file paths must be absolute, relative to “\”. However it is possible to determine the directory where the Ewe application is located by using ewe.io.File.getProgramDirectory() (though this is actually a method in ewe.io.FileBase). You can use the value returned by this as the base directory for saving your application data. This method works on mobile devices and on the desktop.

Simulating a Mobile System

For testing purposes, you can tell the Ewe VM to consider itself as being run on a mobile system, even when running on the desktop, by using the /r option. For example try:

 

ewe /r ewe.ui.Welcome

 

Try it also without the /r to see the difference it makes.

 

You can also specify a /p option to simulate a pen-based system (one without a keyboard). You can also specify a /n option to simulate a system that does not support multiple native windows, but rather a single drawing surface is provided for a running application (e.g. on PalmOS).

Running an Application in Landscape Mode

For some application running on PDAs, it may be preferable to display the application in landscape mode. In this mode, the application is displayed rotated by 90 degrees either clockwise or anti-clockwise.

 

Note that landscape mode is only supported on native Ewe VMs. It is not supported under a Java VM.

 

You can force the VM to display any application in landscape mode by using the /o (rotate) or /O (cOunter-rotate) option. This option must be placed before the class or ewe file specification on the command line. For example:

 

ewe /o ewe.ui.Welcome

 

You can also set your application to run in landscape mode by setting overriding mApp and then setting the static rotated or counterRotated variable to true, before calling setupMainWindow().

 

You can also use this same variable to determine if the application is running in landscape mode.

 

Note that running in landscape mode comes at a significant reduction in graphics performance. However, under fast processors or operating systems it should be acceptable.
