Programming for Portability

This chapters gives information on writing your Ewe programs so that it may be run on a desktop system, mobile system or as an Applet within a browser without alteration. It covers file access, and IO streams.

 

Before we begin there is an important method you should know. ewe.sys.Vm.getAppletProperties(). This returns properties relevant to an Ewe program running as an Applet. Even if you do not use the data returned by this method, you can use it to determine if your program is running as an Application (if this method returns null) or as an Applet (if it returns a PropertyList object).

 

Note that many web servers and operating systems are case-sensitive when dealing with file names, and some are not. You should therefore always assume a case-sensitive system and ensure that the case of the paths you use in your program matches the case of the program stored on your disk. Be wary of Windows! It has a nasty habit of displaying file names with letter cases different to the actual case of the file.

The ewe.io.File Class

The class ewe.io.File provides two functions:

  1. It is used, un-extended, to represent actual files on a file system accessible locally.
  2. It can be extended to represent file-like object on other file systems. Examples of this include ewe.zip.ZipEntryFile, ewe.io.FakeFileSystem.FakeFile and ewex.rapi.RapiFile. These extended File objects can be used by the FileChooser class to browse the file system associated with the extended File object.

Using the un-extended ewe.io.File Class

Using the constructor new File(String path) or new File(File directory,String path) will create a File object that represents a local file. The resulting object cannot be used by an Applet since they are not allowed to access the local file system. You should therefore avoid using this unless you absolutely want to access the local file system.

 

The better way to get a new File object is to call one of the File.getNewFile() methods.

            public static File getNewFile();

            public static File getNewFile(String path);

            public static File getNewFile(File directory, String path);

 

These return a “blank” File object that is of the correct type for use on the default file system. Normally, this will return a standard ewe.io.File object, but there may be circumstances where it does not. For example, if you have a file called “_filesystem.zip” in the working directory of your application when running under Java, or on the server when running as an applet, this zip file is parsed and a “fake” (virtual) file system is created using the file and directory structure within the zip file. Calling File.getNewFile() will then instead return a FakeFileSystem.FakeFile object which is used to access files within this virtual file system.

 

Once you have a File object as provided by getNewFile() you can then use it to access any file in the file system, and even create IO streams to access the data in the files by using the portable file methods as outlined in the section: Portable File Methods.

The Program Directory

It is convenient when storing files associated with an application to be able to determine the “working directory” for the application. On some platforms (e.g. WinCE) there is no concept of a “current working directory” and so all file names must be specified relative to the root directory of a local file volume. One work-around is to have the application put an entry in the system registry that specifies where its data files should be stored. While this can work it can be very inconvenient.

 

For convenience, Ewe provides a way to determine the “program directory” of a running program. If the program is running directly from .class files on the local file system (which can be done on the desktop), and not from a .ewe file, then the program directory will be the current working directory. If the program is running from a .ewe file, then the program directory will be the directory containing the .ewe file. If the program is being run from an Applet, the program directory will be root on the default file system being used (which may be a virtual file system).

 

The program directory can be determined by calling File.getProgramDirectory(). This gets a default File object via File.getNewFile() and then polls it using File.getInfo() to determine the program directory.

Portable File Methods

These are methods that allow you to have full browse and IO access to a file system given a File object appropriate for that system. By using these methods you will never have to use new to acquire a new File object or to acquire IO streams to the files. You will also ensure that your application will work regardless of the underlying file system being used.

 

void set(File directory, String path);

This sets a File to represent a different file object as specified by the parent directory and child path. This is useful if you are processing large directories since it avoids a lot of expensive object creation. The directory provided may be null, but if it is not, then it must be of the same type as the original file.

 

File getNew(File directory, String path);

This is the way you should create a new File object given another one. The newly created object will be of the same class as the original File. The parent directory specified should also be of the same class as the File object being used, or it can be null (in which case path must specify the full path name of the file).

 

File getParent();

This provides the parent File object for the specified object. The returned File will be of the same type as the original.

 

File getChild(String childPath);

This creates a new File object that represents a child file of the original file. The returned File will be of the same type as the original. Note that a File object will always be returned, even if the child does not exist.

 

Stream toWritableStream(boolean append);

This creates (if possible) an output stream that can be used to write to the file represented by the File object. If the file is invalid, or if the file system is read-only, then this will throw an IOException. Note that if this method returns a stream, the file will automatically be created (if it does not exists) and cleared (if it does exist and append is not specified).

 

Stream toReadableStream();

This creates (if possible) an input stream that can be used to read sequentially from the file represented by the File object. If the file does not exist, then this will throw an IOException

 

RandomAccessStream toRandomAccessStream(String mode);

This creates (if possible) a RandomAccessStream that can be used to read and write randomly to the file represented by the File object. The mode should be "r" or "rw". If the mode is not compatible with the file system (e.g. you ask for write access in a read-only system) the File will throw an IOException.

Accessing Program Resources

Program resources are extra read-only files that an application uses during execution. They may reside either on the local file system (when running on a desktop), or on a web server (when running as an Applet) or in a .ewe file (when running on a mobile device or on a desktop). Additionally, in the case of an Applet, the program resources may be stored in a file called “_resources.zip” that is read in once when the Applet initializes.

 

Because of the different ways in which the resources may be stored it is a bad idea to assume that a resource will be in any one particular form. There are two main methods in ewe.sys.Vm which you should use to access the data in a resource.

RandomAccessStream ewe.sys.Vm.openRandomAccessStream(String path, String mode)

This method returns a RandomAccessStream that can be used to access the bytes of the named resource. If the resource is a File on the local file system, then this will return a RandomAccessFile object that can be read or written. Bytes written to the stream will be stored on the local file system in this case. However if the resource is in a .ewe file or resides on the server, then the system will provide a read-only RandomAccessStream. Depending on the platform this RandomAccessStream may allow writes, but in this case the writes will not be saved permanently. The mode should be "r" or "rw".

Stream ewe.sys.Vm.openResource(Class aClass, String resourceName)

This method attempts to open a read-only stream to the data bytes associated with a resource that may be associated with a specific class. If aClass is null or if aClass is a class that has been loaded via the system classloader, then it passes the request on to openRandomAccessStream() and returns the resulting RandomAccessStream. However if aClass has been loaded via a Classloader that has inherited from ewe.util.mClassLoader, then it will call the openResource() method of the mClassLoader to attempt to get a stream to the resource bytes. However, if this fails, it will again pass the request to openRandomAccessStream().

 

Note that as of version 1.14, the int ewe.sys.Vm.loadResourceOrFile(String path, ByteArray destination) method has been deprecated and you should not use it. Instead use openResource().

 

Generally you should only use these methods to get read access to the program resources. If you need to write configuration information you should do so by writing to a File on the local file system.

 

Note that the Vm methods readResource and  releaseResource are used to access system-level, OS dependent resources and not application resources.
