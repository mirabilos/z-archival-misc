Graphics – Drawing and Painting

 

The ewe.fx.Graphics Class

This class provides all the functionality for drawing lines, shapes, text and images onto a drawing surface. Under Ewe, a drawing surface implements the ewe.fx.ISurface interface. In fact this interface has no methods, but merely tells the class library that there exists native methods that allow drawing operations to be performed on the object.

 

Under Ewe there are three types of surfaces:

  1. A Window surface – represented by ewe.ui.Window.
  2. An Image surface – represented by ewe.fx.Image.
  3. A Printer Job surface – represented by ewe.fx.PrinterJob.

 

You create a Graphics object for a surface using the constructor: Graphics(ISurface surface). In fact, this is the only constructor available for Graphics.

Important notes about Graphics Objects

Graphics objects are mostly used in a write only mode. That is, pixel data is sent to the Graphics object only. However there are some Graphics operations that require the ability to read pixel data from the Graphics. This includes:

ü      Copy operations – where an area is copied from an area of a Graphics object to a different area of a possible different Graphics object.

ü      And/Or/XOr drawing operations – where existing pixels are altered by new pixels being written.

ü      Transparency/Alpha-blending operations – where existing pixels are merged somehow with new pixels being written.

 

Although, these operations may not mean that you, the programmer, wish to read the pixel data, they inherently require the underlying graphical system to be able to read pixel data already written to a Graphics object.

 

However this ability is not supported by all systems, on all surface types. When it comes to these types of operations it will generally be the case that:

 

  1. An Image surface will always be able to support pixel reading.
  2. A Window surface may sometimes support pixel reading (e.g. yes on Win32, no on the Zaurus)
  3. A PrinterJob surface will almost never support pixel reading.

 

There are also composite Graphics objects that will not allow such operations. These are Graphics objects that somehow alter the pixel data being written before they turn up on an actual drawing surface. ewe.fx.RotatedGraphics is an example of such a Graphics object.

BufferedGraphics

A BufferedGraphics object can be used to get around the inconsistencies in the capabilities of the different drawing surfaces. Given a target Graphics object, a BufferedGraphics object provides a temporary Image on which you can perform any drawing operation, before the final data is sent to the target Graphics object. The BufferedGraphics object will only provide a temporary image if the target Graphics object does not support the complex graphics operations. Therefore it is safe and efficient to use it, no matter what the surface of the target Graphics happens to be.

 

You should use BufferedGraphics when you are drawing a Control or drawing to a PrinterJob and you need to perform some of these complex operations. You would use it something like this:

 

void doSomeDrawing(Graphics gr)
{
        BufferedGraphics bg = new BufferedGraphics(gr,new Rect(10,10,40,20));
        Graphics draw = bg.getGraphics();
        //…
        // Do all the draw operations now using the "draw" graphics.
        //…
        draw.release();
        gr.flush();
}

 

Drawing and Filling Methods – Pen and Brush Objects

There are a number of drawing operations supported by Graphics, including drawLine(), drawRect(), drawPolygon(), drawEllipse(), drawPie() and drawArc(). Each of these operations use a Pen object to determine the thickness, color and style of the line used. Calling setPen() on the Graphics object will set the Pen to be used for all following draw operations.

 

There are also a number of fill operations supported by Grpahics, including fillRect(), fillPolygon(), fillEllipse(), fillPie()  and fillClosedArc(). Each of these operations draws the outline of the shape using the current Pen object, and then fills it with the current Brush object. Calling setBrush() on the Graphics object will set the Brush to be used for all following fill operations. You can fill an area without drawing the outline by setting the Pen to null by using setPen(null); before the fill operation.

 

There is also a setColor() method that is a simple way of changing the color for subsequent draw operations. setColor() will set the current Pen to be a solid line of thickness one pixel and with the specified color. It will also set the current Brush to be a solid brush of the specified color. There is also a setBackground() method that is used to specify what the current background color should be.

Drawing Images

There are a number of methods that can be used to draw Images, though you should avoid using them if possible. Rather you should use mImage objects whenever you can, and then let the mImage draw itself on the Graphics by calling the mImage.draw(Graphics g, int x, int y, int options) instead. You can create an mImage using a normal image and also specify a transparency mask for the image, or a transparency color for the image. The mImage will then know which method to use to draw itself on the Graphics.

Flushing and Freeing Graphics

Some Graphics objects may cache the draw operations done to them, so that they do not turn up on their destination surface immediately. These Graphics objects will require a flush() method call in order for the surface to be updated to show all the Graphics operations done. A RotatedGraphics object is an example of such a Graphics object.

 

When you are finished using a Graphics object that you have created, you should call free() on it. This calls flush() on the object and then releases all resources associated with the Graphics object. Even thought this will be done when the Graphics object is garbage collected, you should always do this as soon as you can to release shared systems resources that may be used by the object.
